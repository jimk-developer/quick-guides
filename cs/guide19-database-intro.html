🖥️ Multi-User System Architecture

Most real-world business applications don't run on a single machine. They distribute work across a network of hardware so that many users can access shared data at the same time. Understanding this hardware and software stack is the foundation of database programming.

Hardware Components

A typical multi-user (client/server) system has three hardware components:

💻
Clients
Workstations, laptops, phones

🌐

LAN / Internet

🖥

Hosts the database

💻 Clients

User workstations or devices that run the front-end application. They send requests and display results.

🌐 Network

The communication backbone (LAN, WAN, or Internet) that carries data between clients and the server.

🖥 Server

A powerful machine dedicated to storing the database and running the DBMS software that manages it.

Key Term A client/server system divides processing between the clients (presentation + some logic) and the server (data storage + query processing). This is different from a mainframe model, where all processing happens centrally.

🛠️ Software Components

On top of the hardware, several software layers work together to let your C# application communicate with a database:

Layer What it does Examples
Application Software The front-end your users interact with. Requests data via SQL. Your C# Windows Forms app
Data Access API A set of classes that let your code connect to and communicate with a DBMS. ADO.NET, Entity Framework, Dapper
SQL Queries Structured commands sent to the DBMS to retrieve or modify data. SELECT * FROM Customers
DBMS Database Management System — interprets SQL, enforces rules, returns data. SQL Server, MySQL, SQLite
Database The actual stored data: tables, indexes, constraints, and relationships. A .mdf file on SQL Server
Flow Summary Your app → Data Access API → SQL query → DBMS → Database → result set → back to your app. Each layer has a clear responsibility; you typically only write code at the app and SQL layers.

📊 Relational Database Tables

A relational database organizes data into tables. Each table represents one type of entity (customers, products, orders, etc.).

Table Structure

Every table is organized in rows and columns — like a spreadsheet, but with strict rules:

  • Column (field) — one attribute of the entity (e.g., CustomerName)
  • Row (record) — one instance of the entity (e.g., one customer)
  • Primary Key (PK) — a column (or set of columns) that uniquely identifies each row
  • Data type — each column stores one type of data (int, varchar, date, decimal…)

Example — Customers table:

CustomerID (PK) FirstName LastName Email CityID (FK)
1AliceMorganalice@example.com10
2BrianTorresbrian@example.com12
3CarolNguyencarol@example.com10
Primary Key Rule Every table should have a primary key. The DBMS guarantees that no two rows can have the same PK value, so you can always uniquely identify a single row. PK columns are often auto-generated integers (IDENTITY in SQL Server).

🔗 Table Relationships

The "relational" in relational database means tables are linked to each other through foreign keys. A foreign key (FK) is a column in one table whose value matches the primary key of another table.

One-to-Many Relationship (Most Common)

One row in Table A can be related to many rows in Table B, but each row in Table B points back to exactly one row in Table A.

Cities Table

CityID (PK)CityName
10Chicago
12Denver

Customers Table

CustomerID (PK)NameCityID (FK)
1Alice10
2Brian12
3Carol10

City 10 (Chicago) is referenced by two customers — that's the "many" side. Each customer lives in exactly one city — that's the "one" side. Chicago → Alice and Carol is a one-to-many relationship.

Relationship TypeMeaningTypical Example
One-to-Many One parent row, many child rows One customer → many orders
One-to-One One row paired with exactly one other row One employee → one employee detail record
Many-to-Many Many rows in each table linked together (needs a junction table) Many students ↔ many courses

🗄️ SQL Fundamentals

SQL (Structured Query Language) is the universal language for communicating with a relational DBMS. Your C# application builds SQL statements (or uses an ORM that generates them) and sends them to the DBMS via a data access API.

The four core data-manipulation statements are often called CRUD:

🔍 SELECT — Read

Retrieve rows from one or more tables. Returns a result set.

➕ INSERT — Create

Add one or more new rows to a table.

✏️ UPDATE — Modify

Change the values in existing rows.

🗑️ DELETE — Remove

Remove one or more rows from a table.

SELECT — Reading Data

The SELECT statement retrieves rows and columns from a table. The data it returns is called a result set.

-- Retrieve all columns and all rows from Products
SELECT *
FROM   Products

-- Retrieve specific columns and filter rows
SELECT ProductName, Price, Stock
FROM   Products
WHERE  Price < 50

-- Retrieve rows ordered by price, highest first
SELECT ProductName, Price
FROM   Products
WHERE  Stock > 0
ORDER BY Price DESC

SELECT with an INNER JOIN

When data lives in multiple related tables, use a JOIN to combine it in one result set. An INNER JOIN returns only rows where a match exists in both tables.

-- Combine Orders with CustomerName from Customers
SELECT c.CustomerName,
       o.OrderDate,
       o.OrderTotal
FROM   Customers AS c
    INNER JOIN Orders AS o
        ON c.CustomerID = o.CustomerID
WHERE  o.OrderTotal >= 100
ORDER BY c.CustomerName ASC
Aliases make JOINs readable Use AS c and AS o to create short table aliases. Then qualify column names like c.CustomerName to make clear which table each column belongs to — essential when two tables share a column name (e.g., both have CustomerID).

INSERT — Adding a New Row

The INSERT statement adds one new row to a table. You list the columns you want to populate, then the matching values.

-- Add a new product to the Products table
INSERT INTO Products (ProductName, Price, Stock)
VALUES ('Wireless Keyboard', 49.99, 150)
Note on Identity Columns If the table has an auto-increment primary key (e.g., ProductID IDENTITY), you omit it from the column list. The DBMS assigns the next available value automatically.

UPDATE — Modifying Existing Rows

The UPDATE statement changes values in rows that already exist. Always include a WHERE clause — omitting it updates every row in the table.

-- Reduce the price of one specific product
UPDATE Products
    SET   Price = 39.99
WHERE ProductName = 'Wireless Keyboard'

-- Update multiple columns for a row identified by primary key
UPDATE Customers
    SET   Email    = 'new@email.com',
          CityID   = 12
WHERE CustomerID = 1
Always use WHERE with UPDATE An UPDATE without a WHERE clause modifies every row in the table — usually catastrophic. Best practice: identify the target row by its primary key.

DELETE — Removing Rows

The DELETE statement removes one or more rows from a table. Like UPDATE, it requires a careful WHERE clause.

-- Remove a single product by its primary key
DELETE FROM Products
WHERE ProductID = 7

-- Remove all discontinued products (multiple rows)
DELETE FROM Products
WHERE Discontinued = 1
Always use WHERE with DELETE DELETE FROM Products with no WHERE clause deletes all rows from the table. Always target rows precisely — the primary key is the safest filter.

📋 SQL Quick Reference

Statement Purpose Key Clauses
SELECT Read data; returns a result set FROM, WHERE, ORDER BY, INNER JOIN … ON
INSERT INTO … VALUES Add a new row Column list must match values list
UPDATE … SET Modify existing rows WHERE to target rows; SET col = val for each change
DELETE FROM Remove rows WHERE to target rows — omit at your peril!
* wildcard Select all columns Use sparingly in production (wastes bandwidth)
ORDER BY … ASC/DESC Sort the result set ASC (default) = A→Z / 0→9; DESC = reverse
INNER JOIN … ON Combine data from two tables by matching key columns Only returns rows with a match in both tables
