Guide #7: How to Handle Exceptions and Validate Data

Exceptions, Try-Catch, Throw Statements, Data Validation, and Dialog Boxes

💥 What Is an Exception?

An exception is a runtime error — a problem that occurs while your program is running, not while it's compiling. When something goes wrong (like trying to convert the text "abc" to a number), the .NET runtime creates an exception object and throws it. If your code doesn't handle the exception, the application will crash.

Key Terminology

Thrown: When a runtime error occurs, the system creates an exception object and "throws" it — meaning it interrupts normal execution and looks for code that can handle the problem.

Handled (Caught): Your code "catches" the exception using a try-catch statement, allowing the program to respond gracefully instead of crashing.

Unhandled: If no catch block exists for the exception, the program terminates with an error message.

Common Scenarios That Cause Exceptions

🔤 Invalid Format

A user types "hello" into a field that expects a number, and your code tries to convert it with Convert.ToDecimal().

📈 Value Too Large

A calculation produces a number too large to fit in the target data type, like storing 999,999,999,999 in an int.

➗ Division by Zero

Your code attempts to divide an integer by zero, which is mathematically undefined.

🚫 Null Reference

Your code tries to use an object that hasn't been initialized (is null).

🌳 The Exception Hierarchy

All exceptions in C# are objects created from the Exception class or one of its subclasses. This forms a hierarchy — more specific exception types inherit from more general ones.

Exception
↙ ↘
FormatException
ArithmeticException
↙ ↘
OverflowException
DivideByZeroException
...and many more subclasses
Exception Type When It's Thrown
FormatException A value can't be converted to the target type (e.g., converting "abc" to decimal)
OverflowException A value is too large or too small for the target data type
DivideByZeroException An integer division or remainder operation has a divisor of zero
NullReferenceException Code attempts to use a member on an object reference that is null
Exception The base class — can be used to catch any exception type
Why does the hierarchy matter? Because a catch block for Exception will catch any exception (since all exceptions inherit from it), while a catch block for FormatException will only catch format-related errors. You use specific types first and the general Exception type last.

🛡️ Try-Catch Statements

The try-catch statement is the primary mechanism for handling exceptions. You place the code that might throw an exception inside the try block, and you place the code that should run if an exception occurs inside one or more catch blocks.

Syntax: Basic Try-Catch

try
{
    // Code that might throw an exception
}
catch (FormatException ex)
{
    // Runs if a FormatException is thrown
}
catch (OverflowException ex)
{
    // Runs if an OverflowException is thrown
}
catch (Exception ex)
{
    // Runs for any other exception not caught above
}

How Try-Catch Execution Works

When an exception occurs inside the try block, execution immediately jumps to the first matching catch block. The remaining statements in the try block are skipped — they do not execute. If no exception occurs, all the catch blocks are skipped entirely.

Order matters! Place catch blocks from most specific to most general. A catch block for Exception must come last, because it matches every exception type. If you put it first, the more specific catch blocks would never be reached.

Example: Catching a Format Exception

private void btnConvert_Click(object sender, EventArgs e)
{
    try
    {
        decimal temperature = Convert.ToDecimal(txtFahrenheit.Text);
        decimal celsius = (temperature - 32) * 5 / 9;
        lblCelsius.Text = celsius.ToString("f1");
    }
    catch (FormatException)
    {
        MessageBox.Show("Please enter a valid number.", "Entry Error");
        txtFahrenheit.Focus();
    }
}

In this example, if the user types something non-numeric (like "hot"), the Convert.ToDecimal() call throws a FormatException. The calculation line is skipped, and the catch block displays a helpful message.

Catching Multiple Exception Types

try
{
    decimal rate = Convert.ToDecimal(txtRate.Text);
    decimal result = 1000 / rate;
    lblResult.Text = result.ToString("c");
}
catch (FormatException)
{
    MessageBox.Show("Rate must be a number.", "Entry Error");
}
catch (OverflowException)
{
    MessageBox.Show("Rate is too large or too small.", "Entry Error");
}
catch (Exception ex)
{
    MessageBox.Show(ex.Message, "Unexpected Error");
}

🔍 Exception Properties and Methods

When you catch an exception, the exception object provides useful information about what went wrong. You access this information through its properties and methods.

Property / Method What It Returns
Message A human-readable description of the error (e.g., "Input string was not in a correct format.")
StackTrace The list of methods that were called before the exception occurred — extremely useful for debugging
GetType() Returns the specific type of the exception (e.g., FormatException)
ToString() Returns a string that combines the exception type, message, and stack trace

Using Exception Properties

catch (Exception ex)
{
    // Display the error message to the user
    MessageBox.Show(ex.Message, "Error");

    // Get the type of exception
    string type = ex.GetType().ToString();

    // Get the full stack trace for debugging
    string trace = ex.StackTrace;
}
The StackTrace property shows you the exact sequence of method calls that led to the exception. Think of it as a trail of breadcrumbs — it shows where the error originated and how execution got there. This is invaluable when debugging.

🔒 The Finally Block

A try-catch statement can include an optional finally block. The code in the finally block always executes — whether an exception occurred or not, and whether or not a catch block was executed.

Syntax: Try-Catch-Finally

try
{
    // Code that might throw an exception
}
catch (Exception ex)
{
    // Handle the exception
}
finally
{
    // This code ALWAYS runs, no matter what
    // Commonly used for cleanup (closing files, connections, etc.)
}

When to Use Finally

The finally block is most useful when you need to clean up resources — like closing a file, releasing a database connection, or resetting a control's state — regardless of whether the operation succeeded or failed.

🚀 Throw Statements

In addition to catching exceptions, you can also throw exceptions yourself using the throw statement. This is useful when your code detects a condition that shouldn't continue — such as invalid data passed to a method.

Syntax: Throwing an Exception

throw new Exception("Something went wrong.");

// Or throw a more specific exception type
throw new ArgumentException("Value cannot be negative.");

Common Reasons to Throw an Exception

🧪 Testing Exception Handlers

You can throw an exception to verify that your catch blocks are working correctly during development.

🚫 Method Can't Complete

When processing in a method can't be completed (e.g., a required value is missing), throw an exception to signal the problem to the calling code.

🔄 Re-Throwing After Processing

You can catch an exception, perform some processing (like logging), and then throw it again so calling code can handle it too.

Example: Throwing and Re-Throwing

public static decimal CalculateDiscount(decimal price, decimal discountPercent)
{
    if (discountPercent < 0 || discountPercent > 100)
    {
        throw new ArgumentException("Discount must be between 0 and 100.");
    }
    return price * discountPercent / 100;
}

// Re-throwing an exception after logging
catch (Exception ex)
{
    // Log the error, then pass it along
    Console.WriteLine(ex.Message);
    throw;  // re-throws the original exception
}
Use throw; (without an argument) to re-throw the current exception. This preserves the original stack trace. Writing throw ex; resets the stack trace, making debugging harder.

💬 Dialog Boxes with MessageBox

The MessageBox class provides the Show() method, which displays a dialog box to the user. This is commonly used in exception handling to display error messages, and in validation to notify users of invalid entries.

Syntax: MessageBox.Show()

// Simple message
MessageBox.Show("Operation complete.");

// Message with a title
MessageBox.Show("Please enter a valid number.", "Entry Error");

// Message with title, buttons, and icon
DialogResult result = MessageBox.Show(
    "Are you sure you want to exit?",
    "Confirm Exit",
    MessageBoxButtons.YesNo,
    MessageBoxIcon.Question);

if (result == DialogResult.Yes)
{
    this.Close();
}

Common MessageBox Options

Parameter Common Values
MessageBoxButtons OK, OKCancel, YesNo, YesNoCancel, RetryCancel
MessageBoxIcon Information, Warning, Error, Question
DialogResult OK, Cancel, Yes, No, Retry
Using DialogResult: When you use buttons like YesNo, the Show() method returns a DialogResult value that tells you which button the user clicked. You can then use an if statement to respond accordingly.

✅ Data Validation

Data validation is the process of checking user entries before processing them. Instead of relying solely on exception handling to catch bad input, you proactively verify that entries are correct. This provides a better user experience and prevents unnecessary exceptions from being thrown.

Three Types of Validation

When working with text box input in a Windows Forms application, there are three common types of validation you'll perform:

1️⃣ Presence Check

Has the user entered anything? Check that required fields are not empty or whitespace before processing.

2️⃣ Numeric Type Check

Can the entry be converted to the required numeric type? Verify that a text box meant for numbers actually contains a valid number.

3️⃣ Range Check

Is the value within an acceptable range? Confirm that a number falls between a minimum and maximum value (e.g., a quantity must be greater than 0). This type of check can also prevent OverflowException errors.

Why Validate Instead of Just Catching Exceptions?

Exception handling is a safety net, but it shouldn't be your primary strategy for dealing with bad input. Validation gives users specific, helpful feedback about what's wrong (e.g., "Quantity must be greater than 0" instead of a generic error message). It also avoids the performance overhead of throwing and catching exceptions.

Example: Validating Before Processing

private void btnCalculate_Click(object sender, EventArgs e)
{
    // 1. Presence check
    if (txtQuantity.Text == "")
    {
        MessageBox.Show("Quantity is required.", "Entry Error");
        txtQuantity.Focus();
        return;
    }

    // 2. Numeric type check using TryParse
    if (!int.TryParse(txtQuantity.Text, out int quantity))
    {
        MessageBox.Show("Quantity must be a valid integer.", "Entry Error");
        txtQuantity.Focus();
        return;
    }

    // 3. Range check
    if (quantity <= 0 || quantity > 100)
    {
        MessageBox.Show("Quantity must be between 1 and 100.", "Entry Error");
        txtQuantity.Focus();
        return;
    }

    // All validation passed — safe to process
    decimal total = quantity * 19.99m;
    lblTotal.Text = total.ToString("c");
}

🔄 TryParse vs. Convert

There are two common approaches to converting text to a number. Understanding when to use each one is important for writing clean, robust code.

Convert (Throws on Failure)

// Throws FormatException if invalid
decimal price = Convert.ToDecimal(txtPrice.Text);

// Must use try-catch to handle errors

Throws an exception if the text can't be converted. Requires a try-catch to handle gracefully.

TryParse (Returns True/False)

// Returns false if invalid — no exception
if (decimal.TryParse(txtPrice.Text,
    out decimal price))
{
    // price is valid, use it
}

Returns false if conversion fails — no exception is thrown. Cleaner for validation.

Best practice: Use TryParse() when you're validating user input. It's designed specifically for this purpose — it attempts the conversion and returns true or false instead of throwing an exception. Use Convert methods when you're confident the data is valid (like data from a database or after validation has already passed).

♻️ Generic Validation Methods

When a form has multiple text boxes that all need similar validation, writing the same validation code over and over creates duplication. Instead, you can write generic validation methods — reusable methods that validate one entry at a time and can be called for each field.

Why Write Generic Methods?

Generic validation methods let you reuse your validation code across multiple fields and even across multiple forms. Instead of repeating the same checks for every text box, you write the logic once and call it as needed.

Example: Reusable Validation Methods

// Returns true if the text box has an entry
public bool IsPresent(TextBox textBox, string name)
{
    if (textBox.Text == "")
    {
        MessageBox.Show(name + " is a required field.", "Entry Error");
        textBox.Focus();
        return false;
    }
    return true;
}

// Returns true if the text box contains a valid decimal
public bool IsDecimal(TextBox textBox, string name)
{
    if (!decimal.TryParse(textBox.Text, out _))
    {
        MessageBox.Show(name + " must be a valid decimal.", "Entry Error");
        textBox.Focus();
        return false;
    }
    return true;
}

// Returns true if the value is within the specified range
public bool IsWithinRange(TextBox textBox, string name,
    decimal min, decimal max)
{
    decimal value = Convert.ToDecimal(textBox.Text);
    if (value < min || value > max)
    {
        MessageBox.Show(name + " must be between " +
            min + " and " + max + ".", "Entry Error");
        textBox.Focus();
        return false;
    }
    return true;
}

Two Ways to Use Generic Validation Methods

Once you have reusable validation methods, there are two common patterns for calling them in a method that validates all entries on a form:

Approach 1: Sequential with Early Return

private bool IsValidData()
{
    if (!IsPresent(txtPrice, "Price"))
        return false;
    if (!IsDecimal(txtPrice, "Price"))
        return false;
    if (!IsPresent(txtQty, "Quantity"))
        return false;
    return true;
}

Stops at the first error and shows one message at a time. User fixes one issue, then sees the next (if any).

Approach 2: Boolean AND Chain

private bool IsValidData()
{
    return
        IsPresent(txtPrice, "Price") &&
        IsDecimal(txtPrice, "Price") &&
        IsPresent(txtQty, "Quantity") &&
        IsDecimal(txtQty, "Quantity");
}

Uses short-circuit evaluation — the && operator stops evaluating as soon as one method returns false.

Both approaches work the same way thanks to short-circuit evaluation. The && operator won't evaluate the right side if the left side is false, so validation stops at the first failure in either approach.

Calling the Validation Method

private void btnCalculate_Click(object sender, EventArgs e)
{
    if (IsValidData())
    {
        // All validation passed — safe to process
        decimal price = Convert.ToDecimal(txtPrice.Text);
        int qty = Convert.ToInt32(txtQty.Text);
        decimal total = price * qty;
        lblTotal.Text = total.ToString("c");
    }
}
Notice: Because IsValidData() has already confirmed the entries are present, numeric, and in range, the Convert calls inside the if block are guaranteed to succeed — no try-catch needed for those conversions!

🧩 Putting It All Together

In practice, you'll often use both validation and exception handling. Validation handles the expected cases (missing entries, bad formats, out-of-range values), while exception handling serves as a safety net for anything unexpected.

Recommended Approach

private void btnCalculate_Click(object sender, EventArgs e)
{
    try
    {
        if (IsValidData())
        {
            decimal price = Convert.ToDecimal(txtPrice.Text);
            int qty = Convert.ToInt32(txtQty.Text);
            decimal total = price * qty;
            lblTotal.Text = total.ToString("c");
        }
    }
    catch (Exception ex)
    {
        MessageBox.Show(ex.Message, "Unexpected Error");
    }
}

The Two-Layer Defense

Layer 1 — Validation: Catches expected input problems (empty fields, non-numeric text, out-of-range values) and provides specific, user-friendly messages.

Layer 2 — Exception Handling: Catches anything unexpected that slips through, preventing the application from crashing.
