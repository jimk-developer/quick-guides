A Student's Guide to Writing Clean, Professional Code

This guide combines industry-accepted practices from Microsoft, Google, and established C# programming resources. Master these standards now, and you'll write better code throughout your career.

üìù Naming Conventions

Good names make your code self-documenting. The name of a variable, method, or class should tell the reader exactly what it represents or does.

The Two Main Casing Styles

PascalCase

First letter of every word is capitalized.

Used for: Classes, Methods, Properties, Constants, Namespaces, Enums

CalculateTotal
CustomerName
MaxItemCount

camelCase

First word lowercase, then capitalize first letter of subsequent words.

Used for: Local variables, Parameters, Private fields

totalAmount
customerName
itemCount

Choosing Good Names

Use meaningful, descriptive names that explain the purpose of the variable or method. The name should answer "What does this represent?" or "What does this do?"
Use single letters (except for loop counters), abbreviations that aren't universally understood, or names that don't reflect the purpose.

‚úÖ Good Names

int numberOfSheep;
decimal totalPrice;
string customerName;
bool isValidInput;
double widthInMeters;

‚ùå Bad Names

int n;
decimal x;
string s1;
bool flag;
double w;
If you can't think of a good name, you probably don't understand what the variable represents. Step back and think about its purpose before naming it.

üìê Code Formatting

Well-formatted code is easier to read, debug, and maintain. Consistent formatting shows professionalism and helps others understand your code quickly.

Indentation

  • Use 4 spaces for each level of indentation (Visual Studio default)
  • Code inside braces { } should be indented one level
  • Nested structures should be indented at each level

‚úÖ Good Indentation

if (isValid)
{
    for (int i = 0; i < count; i++)
    {
        ProcessItem(i);
    }
}

‚ùå Bad Indentation

if (isValid)
{
for (int i = 0; i < count; i++)
{
ProcessItem(i);
}
}

Braces

üìè The Brace Rules

  • Always use braces ‚Äî even for single-statement if/for/while blocks
  • Opening brace goes on its own line (Allman style, C# standard)
  • Closing brace goes on its own line, aligned with the opening keyword

‚úÖ Always Use Braces

if (score >= 60)
{
    lblResult.Text = "Pass";
}

‚ùå Skipping Braces

if (score >= 60)
    lblResult.Text = "Pass";
// Easy to add bugs later!

Spacing

  • One space after keywords: if (, for (, while (
  • One space after commas: Method(a, b, c)
  • Spaces around binary operators: x + y, a == b
  • No space after unary operators: i++, !isValid
  • No space inside parentheses: (x + y) not ( x + y )

Blank Lines

  • Use blank lines to separate logical groups of statements
  • One blank line between methods
  • Two blank lines between class sections (fields, properties, methods)

Line Length

Keep lines under 100 characters when possible. Long lines are hard to read and may require horizontal scrolling. Break long statements across multiple lines. 
// Long line - hard to read
Console.WriteLine("The total price including tax is: " + totalPrice.ToString("c") + " for customer: " + customerName);

// Better - broken across lines
Console.WriteLine(
    "The total price including tax is: " + 
    totalPrice.ToString("c") + 
    " for customer: " + customerName);

üí¨ Comments & Documentation

Comments explain why code does something, not what it does. Good code with good names should be self-explanatory for the "what."

Types of Comments

File Header Comments

Every source file should start with a header comment identifying the program, author, and purpose:

/*
 * Program:     Double Glazing Calculator
 * Author:      Your Name
 * Date:        February 2026
 * Purpose:     Calculate wood and glass requirements for windows
 * 
 * Modification History:
 * Date         Author      Description
 * ----------   --------    ----------------------------------
 * 02/02/2026   YourName    Initial version
 */

Good vs Bad Comments

‚úÖ Good Comments

// Add 99 before dividing to round up
// (ensures any partial bottle counts as full)
int bottles = (tablets + 99) / 100;

// Convert meters to feet (3.25 ft/m)
woodLength = meters * 3.25;

‚ùå Bad Comments

// Add one to count
count = count + 1;

// Set x to 5
x = 5;

// This is a variable
int total;
Comment on the "why" ‚Äî business rules, algorithms, workarounds, or anything that isn't obvious from reading the code.
Don't state the obvious. If your code needs a comment to explain what it does, consider renaming variables or simplifying the logic instead.
Comments can become outdated if not maintained. When you change code, update the related comments too!

üì¶ Variables & Constants

Declaration Best Practices

  • Declare variables as close to their first use as possible
  • Initialize variables when you declare them when possible
  • One declaration per line for clarity
  • Use the most restrictive type that meets your needs

‚úÖ Good Declarations

int customerCount = 0;
decimal totalPrice = 0.00m;
string customerName = "";
bool isValid = false;

‚ùå Bad Declarations

int a, b, c, d, e, f;
decimal x;
string s1, s2;

Constants

Use constants for values that never change. This makes your code more readable and easier to maintain.

Use named constants instead of "magic numbers" ‚Äî unexplained numeric values scattered through your code.

‚úÖ Using Constants

const decimal TaxRate = 0.075m;
const double FeetPerMeter = 3.25;
const int MaxItems = 100;

decimal tax = subtotal * TaxRate;
double feet = meters * FeetPerMeter;

‚ùå Magic Numbers

decimal tax = subtotal * 0.075m;
double feet = meters * 3.25;

// What do these numbers mean?
// What if the tax rate changes?

  • Use PascalCase for method names
  • Start with a verb that describes what the method does
  • Be descriptive but not overly long

Event Handler Naming

Visual Studio generates event handlers with the pattern: ControlName_EventName

// Button click handlers
private void btnCalculate_Click(object sender, EventArgs e)
{
    // Calculate and display results
}

private void btnClear_Click(object sender, EventArgs e)
{
    // Clear all input fields
}

private void btnExit_Click(object sender, EventArgs e)
{
    this.Close();
}

Method Organization

üìè Method Should Do One Thing

Each method should have a single, clear purpose. If you can't describe what a method does in one sentence without using "and," it probably does too much.

Keep methods short and focused. A method longer than 20-30 lines might need to be split into smaller methods.

üîÄ Control Structures

if Statements

// Simple if
if (score >= 60)
{
    lblResult.Text = "Pass";
}

// if-else
if (score >= 60)
{
    lblResult.Text = "Pass";
}
else
{
    lblResult.Text = "Fail";
}

// if-else if-else chain
if (score >= 90)
{
    grade = "A";
}
else if (score >= 80)
{
    grade = "B";
}
else if (score >= 70)
{
    grade = "C";
}
else
{
    grade = "F";
}

switch Statements

switch (dayNumber)
{
    case 1:
        dayName = "Monday";
        break;
    case 2:
        dayName = "Tuesday";
        break;
    case 3:
        dayName = "Wednesday";
        break;
    default:
        dayName = "Unknown";
        break;
}

Loops

// for loop - when you know the count
for (int i = 0; i < items.Length; i++)
{
    ProcessItem(items[i]);
}

// foreach loop - for reading collections
foreach (string name in names)
{
    Console.WriteLine(name);
}

// while loop - condition checked first
while (count < maxCount)
{
    count++;
}

// do-while loop - always executes at least once
do
{
    input = GetUserInput();
} while (!IsValid(input));
Use for loops when you know the number of iterations. Use foreach when iterating through a collection without needing the index. Use while when you don't know how many iterations are needed.

üìä Arrays & Collections

Array Naming

  • Use plural nouns for arrays and collections
  • The name should describe what the array contains

‚úÖ Good Array Names

int[] scores;
string[] customerNames;
decimal[] prices;
List<string> items;

‚ùå Bad Array Names

int[] arr;
string[] data;
decimal[] nums;
List<string> list1;

Array Initialization

// Declare with size
int[] scores = new int[10];

// Declare and initialize
string[] colors = { "Red", "Green", "Blue" };

// Decimal arrays need 'm' suffix
decimal[] prices = { 19.99m, 29.99m, 39.99m };

// 2D array
int[,] grid = new int[3, 4];  // 3 rows, 4 columns

List<T> Best Practices

// Prefer List when size can change
List<string> customers = new List<string>();

// Add items
customers.Add("Alice");
customers.Add("Bob");

// Use Count (not Length) for lists
int total = customers.Count;
Prefer List<T> over arrays when the size can change. Use arrays when the size is fixed and known at compile time.

The Golden Rules

1. Be Consistent

Pick a style and stick with it throughout your entire project. Inconsistent code is harder to read than code with a "wrong" but consistent style.

2. Write for the Reader

Code is read far more often than it's written. Optimize for readability, not for typing speed. The person reading your code might be you, six months from now.

3. Keep It Simple

The simplest solution that works is usually the best. Don't add complexity unless it provides clear value.

4. Fix Problems Early

Address warnings and fix small issues immediately. Technical debt accumulates quickly.

5. Test As You Go

Test your code frequently. Small, incremental changes are easier to debug than large ones.

Before Submitting Code

  • ‚úÖ All controls have meaningful names with correct prefixes
  • ‚úÖ All variables have descriptive names using camelCase
  • ‚úÖ No magic numbers ‚Äî use named constants
  • ‚úÖ Code is properly indented
  • ‚úÖ All braces are present and aligned
  • ‚úÖ File has a header comment with your name and purpose
  • ‚úÖ Complex logic has explanatory comments
  • ‚úÖ No compiler warnings
  • ‚úÖ Code has been tested with valid and invalid input
Professional programmers know: The time spent following coding standards is always less than the time spent debugging poorly written code.
