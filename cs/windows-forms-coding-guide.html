Object-Oriented Programming, Events, and Debugging in C#

üéØ Object-Oriented Programming Concepts

C# is an object-oriented programming language. Understanding these fundamental concepts is essential for writing Windows Forms applications.

üì¶ Class

Code that defines the characteristics of an object. Think of it as a blueprint or template that specifies what properties and methods an object will have.

üî∑ Object

An instance of a class. When you create an object from a class, you're creating a specific item based on that blueprint.

‚ö° Instantiation

The process of creating an object from a class. When you instantiate a class, you create a new instance of that class.

üìã Instance

A specific object created from a class. Each instance is independent and has its own set of property values.

Class Members

A class defines several types of members that give objects their capabilities:

Member Type Description Example
Property A characteristic of an object and its associated data txtName.Text
Method An operation that an object can perform txtName.Focus()
Event A signal that something has happened (user action or system occurrence) Click, Load
Remember: Properties, methods, and events are all members of a class. "Notices" are NOT defined by a class‚Äîthey don't exist in C# programming!

üîó Accessing Object Members

To refer to a property or method of an object in C#, use dot notation: the object name, followed by a period (dot), followed by the property or method name.

Syntax for Properties

// Getting a property value
string name = txtName.Text;

// Setting a property value
lblResult.Text = "Hello!";

Syntax for Methods

// Calling a method with no parameters
txtMonthlyInvestment.Focus();

// Calling a method to close the current form
this.Close();

The this Keyword

The this keyword refers to the current instance of the class (the current form). Use this.Close() to close the current form.

üé™ Events and Event Handlers

A Windows Forms application is event-driven, meaning it responds to events. These include both user events (like clicking a button) and other types of events (like the form loading).

What is an Event Handler?

An event handler is a method that responds to events. When an event occurs, the corresponding event handler code executes.

Creating Event Handlers

The easiest way to create an event handler is to double-click a control in the Form Designer:

  1. Double-click the control in the Form Designer (e.g., a button)
  2. Visual Studio generates the event handler skeleton code automatically
  3. Add your code inside the curly braces of the event handler
Default Event: When you double-click a Button, Visual Studio generates code for the Click event (not DoubleClick!). The Click event is the button's default event.

Event Handler Naming Convention

Visual Studio automatically names event handlers using this pattern:

// Pattern: ControlName_EventName
private void btnCalculate_Click(object sender, EventArgs e)
{
    // Your code goes here
}

// Another example
private void btnExit_Click(object sender, EventArgs e)
{
    this.Close();
}

üìù C# Syntax Essentials

Statements and Blocks

Element Symbol Description
Statement terminator ; (semicolon) Every C# statement must end with a semicolon
Code blocks { } (braces) Blocks of code are enclosed in curly braces
Method calls ( ) (parentheses) Method parameters are enclosed in parentheses
Member access . (period/dot) Separates object name from property/method

Code Example with Syntax Elements

private void btnCalculate_Click(object sender, EventArgs e)
{                                                    // Opening brace starts block
    decimal total = 100.50m;                       // Statement ends with semicolon
    string amount = total.ToString("c");           // Method call with parameter
    lblResult.Text = amount;                        // Property assignment
}                                                    // Closing brace ends block

‚ú® Code Formatting Best Practices

Well-formatted code is easier to read, understand, and maintain. Follow these guidelines:

üìè Indentation

Use indentation to align related elements of code. Code inside braces should be indented one level.

üìÑ Blank Lines

Use blank lines before and after groups of related statements to visually separate logical sections.

‚ÜîÔ∏è Spacing

Use spaces to separate words, values, and operators for better readability.

Avoid: Using all capital letters for variable names is NOT a recommended practice for code readability in C#.

üí¨ Comments

Comments document your code and are ignored by the compiler. They don't affect how your application runs.

Single-Line Comments

// This is a single-line comment
string amount = total.ToString("c");   // currency format

Multi-Line Comments

/*
 * This is a multi-line comment
 * It can span multiple lines
 * Use it for longer explanations
 */

‚ö° Comment Quick Reference

Type Syntax Use For
Single-line // comment Brief notes, end-of-line explanations
Multi-line /* comment */ Longer explanations, file headers
XML Documentation /// comment Method/class documentation
Important: Comments can appear at the end of a line or on their own line. They can contain any text, including C# keywords. However, comments can become inaccurate or out of date if not maintained‚Äîalways update comments when you change code!

Refactoring

Refactoring is the process of restructuring existing code without changing its behavior. Visual Studio provides built-in refactoring tools.

Rename Refactoring: To change the name of all occurrences of a variable after you change the first occurrence, use refactoring. Right-click the variable and select Rename (or press Ctrl+R, Ctrl+R).

üêõ Understanding Errors

There are two main types of errors you'll encounter when developing C# applications:

Aspect Syntax Error (Build Error) Runtime Error (Exception)
When Detected While you type or when you build When the application runs
Cause Violates C# language rules (typos, missing semicolons) Problem prevents statement execution (divide by zero, invalid input)
Where Shown Code Editor (red squiggle) AND Error List window Exception dialog at runtime
Can App Run? No‚Äîmust fix before running App starts but crashes when error occurs
Also Called Build error, compile error Exception

Syntax Error Example

// Missing semicolon - syntax error!
string name = txtName.Text    // ‚ùå Error: ; expected

// Corrected
string name = txtName.Text;   // ‚úÖ Correct

Runtime Error Example

// This compiles fine but crashes if user enters non-numeric text
decimal amount = Convert.ToDecimal(txtAmount.Text);  // ‚ùå Runtime error if text isn't a number

üîç Testing vs. Debugging

Testing and debugging are two distinct but related activities:

üß™ Testing

Goal: Find errors

Running the application with various inputs to discover runtime errors and verify it works correctly.

üîß Debugging

Goal: Fix errors

Finding the cause of errors and correcting them. Involves examining code and variable values.

Remember: Testing = Finding bugs | Debugging = Fixing bugs

üõ†Ô∏è Debugging Tools

Data Tips

A data tip displays the value of a variable or property when you hover over it during debugging. This helps you understand what values your code is working with at runtime.

Using Data Tips

  • Set a breakpoint by clicking in the left margin of the Code Editor
  • Run the application in debug mode (F5)
  • When execution pauses at the breakpoint, hover over any variable
  • The data tip shows the current value of that variable

Other Debugging Tools
