Understanding How JavaScript Moves Declarations to the Top

ğŸ¯ What is Hoisting?

Hoisting is JavaScript's default behavior of moving declarations to the top of their scope before code execution. It's as if JavaScript "lifts" your variable and function declarations to the top of the current scope (script or function).

Think of it like a classroom where the teacher takes attendance at the start of class. Before any lesson begins, the teacher knows who's supposed to be thereâ€”even if some students haven't arrived yet. JavaScript does something similar: it scans your code and "registers" all declarations before running any code.

ğŸ’¡ Key Point: Only declarations are hoisted, not initializations (the values assigned to variables). This is a crucial distinction that causes many bugs!

A Simple Example

// What you write:
console.log(message);  // What happens here?
var message = "Hello!";

// What JavaScript sees (after hoisting):
var message;           // Declaration hoisted to top
console.log(message);  // Output: undefined (not an error!)
message = "Hello!";    // Initialization stays in place
ğŸ“Œ Remember: The variable message exists (is declared) from the start, but it doesn't have a value until the assignment line runs. That's why we get undefined instead of an error.

ğŸ” What Hoisting Really Means

When we say declarations are "hoisted," it does not mean the JavaScript engine rewrites your code or physically moves lines of text to the top of the file.

Instead, it means that memory is allocated for variables and functions before the code is actually executed. The code stays exactly where you wrote itâ€”JavaScript just knows about your declarations ahead of time.

âš ï¸ Common Misconception: Many tutorials show hoisting as if JavaScript literally moves your code around. This is a helpful mental model, but it's not what actually happens. Your code stays in place; JavaScript just processes it in two phases.

âŒ What Hoisting is NOT

JavaScript does NOT rewrite your code like this:

// Your code is NOT changed to:
var x;  // Moved up?
console.log(x);
x = 5;

âœ… What Hoisting IS

JavaScript allocates memory before running:

// Your code stays as-is:
console.log(x);
var x = 5;
// But JS already knows x exists

âš™ï¸ The Two Phases of Execution

To understand hoisting, you need to know that JavaScript processes your code in two distinct phases. This two-phase process is the key to understanding why hoisting works the way it does.

Phase 1: The Creation Phase (Memory Allocation)

Before running any code, JavaScript scans through your entire script and allocates memory for all declarations it finds:

  • Variables declared with var are registered and set to undefined
  • Variables declared with let/const are registered but left uninitialized (TDZ)
  • Function declarations are registered with their entire function body
  • No code is executed yetâ€”this is just preparation

Phase 2: The Execution Phase

Now JavaScript runs your code line by line, from top to bottom:

  • Assignments give values to variables (x = 5)
  • Function calls are executed
  • Expressions are evaluated
  • Your program logic runs in order

ğŸ“¦ Variable Hoisting with var

Variables declared with var are hoisted to the top of their function scope (or global scope if outside any function). They are initialized with undefined.

Basic var Hoisting

// Example 1: Accessing before declaration
console.log(x);  // Output: undefined (NOT an error)
var x = 5;
console.log(x);  // Output: 5

// Example 2: Multiple declarations
console.log(a, b, c);  // Output: undefined undefined undefined
var a = 1;
var b = 2;
var c = 3;
console.log(a, b, c);  // Output: 1 2 3

var is Function-Scoped

function example() {
    console.log(message);  // Output: undefined
    var message = "Hello";
    console.log(message);  // Output: Hello
}

example();

// The function above is interpreted as:
function example() {
    var message;           // Hoisted to top of function
    console.log(message);  // undefined
    message = "Hello";     // Assignment stays here
    console.log(message);  // Hello
}

var Ignores Block Scope

function test() {
    console.log(x);  // Output: undefined (hoisted!)
    
    if (true) {
        var x = 10;   // var ignores the if block
    }
    
    console.log(x);  // Output: 10 (accessible outside if block!)
}

test();

// var is hoisted to the function scope, not block scope:
function test() {
    var x;              // Hoisted here (function top)
    console.log(x);     // undefined
    
    if (true) {
        x = 10;
    }
    
    console.log(x);     // 10
}
âš ï¸ Problem with var: Because var ignores block scope, it can lead to unexpected behavior in loops and conditionals. This is one of the main reasons let and const were introduced in ES6.

ğŸš« let and const: The Temporal Dead Zone

Variables declared with let and const are also hoisted, but they behave differently. They are not initialized until their declaration is reached. Accessing them before declaration causes a ReferenceError.

ğŸ•³ï¸ Temporal Dead Zone (TDZ): The period between entering a scope and the variable's declaration where the variable exists but cannot be accessed. Any attempt to read or write the variable in this zone throws an error.

Comparing var, let, and const

// var - hoisted and initialized to undefined
console.log(a);  // Output: undefined
var a = 1;

// let - hoisted but NOT initialized (TDZ)
console.log(b);  // âŒ ReferenceError: Cannot access 'b' before initialization
let b = 2;

// const - hoisted but NOT initialized (TDZ)
console.log(c);  // âŒ ReferenceError: Cannot access 'c' before initialization
const c = 3;

ğŸ”§ Function Hoisting

Function declarations are fully hoistedâ€”both the name and the function body. This means you can call a function before it appears in your code!

Function Declarations are Fully Hoisted

// âœ… This works! Function declarations are fully hoisted
sayHello();  // Output: Hello!

function sayHello() {
    console.log("Hello!");
}

// You can call the function anywhere in the same scope
sayHello();  // Output: Hello!

Function Expressions are NOT Hoisted

When you assign a function to a variable (function expression), only the variable declaration is hoisted, not the function itself. 

// âŒ This does NOT work!
greet();  // TypeError: greet is not a function

var greet = function() {
    console.log("Hi!");
};

// Why? JavaScript sees it as:
var greet;           // Declaration hoisted (greet = undefined)
greet();             // âŒ Trying to call undefined as a function!
greet = function() {
    console.log("Hi!");
};

Arrow Functions are NOT Hoisted

// âŒ Arrow functions behave like function expressions
add(2, 3);  // ReferenceError: Cannot access 'add' before initialization

const add = (a, b) => a + b;

// âœ… Correct: declare before calling
const add = (a, b) => a + b;
console.log(add(2, 3));  // Output: 5

âœ… Hoisted (Can Call Before Declaration)

// Function Declaration
sayHi();  // Works!

function sayHi() {
    console.log("Hi!");
}

âŒ NOT Hoisted (Must Declare First)

// Function Expression
sayHi();  // Error!

const sayHi = function() {
    console.log("Hi!");
};

ğŸ“ Class Hoisting

Classes in JavaScript are also hoisted, but like let and const, they are in the Temporal Dead Zone until the declaration is reached. 

// âŒ Cannot use class before declaration

ğŸ“Œ Note: Both class declarations and class expressions are subject to the TDZ. Always define your classes before using them.

ğŸ”² Hoisting in Different Scopes

Hoisting occurs within the current scope. Variables and functions are hoisted to the top of their containing scope, not to the top of the entire program.

Global Scope

console.log(globalVar);  // undefined (hoisted to top of script)

globalFunc();  // Works! (hoisted to top of script)
function globalFunc() {
    console.log("I'm a global function");
}

    console.log(innerVar);  // undefined (hoisted within outer)
    var innerVar = "I'm inside outer";
    
    innerFunc();  // Works! (hoisted within outer)
    function innerFunc() {
        console.log("I'm an inner function");
    }
}

outer();

// innerVar and innerFunc are NOT accessible here
// console.log(innerVar);  // ReferenceError
// innerFunc();            // ReferenceError

    // TDZ for blockVar starts here

ğŸš« Common Hoisting Pitfalls

1. Assuming Variables Have Values Before Assignment

var price = 100;

function calculateTax() {
    console.log(price);  // âŒ undefined, NOT 100!
    var price = 200;      // This shadows the outer price
    console.log(price);  // 200
}

calculateTax();

// Why? The inner 'var price' is hoisted to the top of the function

2. Using var in Loops

// Classic problem with var in loops
for (var i = 0; i < 3; i++) {
    setTimeout(() => console.log(i), 100);
}
// Output: 3, 3, 3 (not 0, 1, 2!)
// Why? var is function-scoped, so there's only ONE 'i' variable

// âœ… Fix: Use let (block-scoped)
for (let i = 0; i < 3; i++) {
    setTimeout(() => console.log(i), 100);
}
// Output: 0, 1, 2 (correct!)

3. Function Expression Before Declaration

// âŒ Common mistake
const result = multiply(5, 3);  // ReferenceError!

const multiply = (a, b) => a * b;

// âœ… Fix: Define function before using
const multiply = (a, b) => a * b;

const result = multiply(5, 3);  // 15

4. Re-declaring Variables with var

var name = "Alice";
var name = "Bob";     // No error! Silently overwrites
console.log(name);    // Output: Bob

// âœ… With let, this throws an error (much safer!)
let age = 25;
let age = 30;         // âŒ SyntaxError: 'age' has already been declared

ğŸ’¡ Best Practices

âœ… Follow These Guidelines:
  1. Use const by default - Only use let when you need to reassign
  2. Avoid var - It has confusing scoping behavior due to hoisting
  3. Declare variables at the top - Makes code clearer and matches hoisted behavior
  4. Declare functions before calling them - Even though declarations are hoisted, it's clearer
  5. Use function declarations for named functions - Use arrow functions for callbacks
  6. Initialize variables when declaring - Avoid let x; without a value when possible

Good Code Organization

// âœ… Good: Declarations at the top, clear and organized
const TAX_RATE = 0.08;
let totalPrice = 0;

function calculateTotal(price, quantity) {
    const subtotal = price * quantity;
    const tax = subtotal * TAX_RATE;
    return subtotal + tax;
}

totalPrice = calculateTotal(29.99, 3);
console.log(totalPrice);  // Output: 97.17
// âŒ Bad: Scattered declarations, relies on hoisting
calculateTotal(29.99, 3);

    var subtotal = price * quantity;
    console.log(tax);  // undefined - confusing!
    var tax = subtotal * TAX_RATE;
