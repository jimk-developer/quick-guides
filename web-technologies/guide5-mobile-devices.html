📱 The Mobile Reality

For most of the early web's history, websites were built almost exclusively for desktop computers. Then smartphones arrived. Today, more than half of all web traffic comes from mobile devices, and for many sites the majority of visitors are on a phone.

A site that only works well on a 27-inch monitor is failing most of its audience. The job of a modern web designer is to build pages that look good and work smoothly on every screen, from a 4-inch phone to a 32-inch display.

Two Approaches to Mobile

There are two strategies for serving mobile visitors: build a separate mobile website, or design one responsive website that adapts to every screen.

❌ Separate Mobile Website

A second site at a different address (often m.example.com) built specifically for phones. Visitors get redirected based on their device.

Drawbacks: Two sites to maintain, two URLs to remember, content can drift between them, and converting an existing large site to a separate mobile version is slow.

✅ Responsive Design

One website with one URL. The same HTML and CSS adjust their layout based on the visitor's screen size. The page rearranges itself.

Benefits: One codebase to maintain, one URL for sharing and search rankings, content stays in sync, and the design naturally accommodates new device sizes.

Responsive design is the modern standard. Almost every website built today uses responsive techniques. Separate mobile sites are now considered a legacy approach.

🏛️ The Three Pillars of Responsive Design

Responsive design isn't a single feature. It's a combination of three CSS techniques that work together.

💧 Fluid Layouts

Layout dimensions use relative units (percentages, rem, em, vw) instead of fixed pixel values. Columns expand and contract with the viewport.

🖼️ Responsive Images

Images grow and shrink with their container instead of having fixed dimensions. One short CSS rule does most of the work.

🔍 Media Queries

CSS rules that detect the device's media type or capability and apply different styles based on screen width or other conditions.

Use all three together and a single page rearranges itself for whatever device the visitor is using.

🪟 The Viewport

The viewport is the visible area of a webpage inside the browser window. The word "viewport" is preferred over "window" because mobile browsers and embedded browsers don't always have a traditional window. The viewport is whatever rectangular area is currently displaying the page.

The viewport is the visible area of the page

The Meta Viewport Element

By default, mobile browsers assume a webpage was designed for a desktop screen. They render the page at a desktop width and then shrink it to fit the phone. The result is text and tap targets that are too small to use comfortably.

To prevent this, every responsive HTML page must include a meta viewport element in the <head>. This single line tells the browser to render the page at the device's actual width.

The Meta Viewport Element

<meta name="viewport"
      content="width=device-width, initial-scale=1.0">
Setting What It Does
width=device-width Sets the viewport width to match the actual screen width of the device.
initial-scale=1.0 Starts the page at 100% zoom level when first loaded.
Don't skip this. Without the meta viewport element, your responsive CSS won't work on mobile devices. The browser will pretend it has a desktop-sized screen and shrink everything down. Add it to every HTML file before doing anything else.

🚀 The Mobile-First Strategy

The mobile-first strategy is exactly what it sounds like: design for the smallest screen first, then layer on enhancements for larger screens. It is the opposite of starting with a desktop design and trying to cram it into a phone.

Why Mobile First Works

🎯 Forces Prioritization

A small screen has little room. Designing it first forces you to decide what content really matters and to leave behind anything that doesn't.

⚡ Better Performance

Mobile devices often have slower connections. Building lean from the start means a faster site for everyone.

🪜 Progressive Enhancement

Adding features for bigger screens is easier than removing features for smaller ones. Start simple and grow.

📊 Where Visitors Live

Mobile traffic dominates most sites. Designing for the majority of your audience first is just good practice.

Mobile-First in Five Steps

1
Add the meta viewport element to every HTML file.
2
Write base styles for mobile. Use a single column. Make text large enough to read. Make tap targets large enough to tap.
3
Test in a mobile viewport using your browser's device mode or a mobile simulator.
4
Add media queries with progressively larger breakpoints, expanding the layout for tablets, then desktops.
5
Test on real devices. Simulators are good, but actual phones reveal issues simulators miss.
Mobile-first design principles: a single-column layout to avoid horizontal scrolling, large touchable buttons, plenty of spacing between tap targets, readable text at default zoom, and an ideal page that doesn't require any scrolling at all on common phone screens.

📐 CSS Units of Measurement

Responsive sites depend on relative units of measurement. Where a fixed-pixel design says "this column is exactly 300 pixels wide", a fluid design says "this column is 50% of its parent" or "this column is 20rem wide". Relative units adjust automatically.

Absolute Units

Absolute units are fixed sizes that never change regardless of context.

Unit Name Use For
px Pixels Borders, small fixed details. The most common absolute unit on the web.
pt Points Print stylesheets only. Avoid for screen.
cm / mm / in Centimeters / millimeters / inches Print stylesheets. Rarely useful on screen.

Relative Units

Relative units scale based on something else: the parent element's size, the root element's size, or the viewport's size. These are the workhorses of responsive design.

Unit Relative To Common Use
% The parent element's size Column widths, container widths
em The current element's font size Padding and margins relative to text size
rem The root element's font size (usually 16px) Font sizes, spacing throughout the site
vw / vh 1% of viewport width / height Hero sections, full-screen elements
Rule of thumb: use rem for font sizes and spacing, % for column widths, and px only for fine details like 1-pixel borders.

💧 Fluid Layouts

A fluid layout uses percentage-based widths so columns expand and contract with the viewport. A fixed layout uses pixel widths and stays the same size regardless of screen.

📏 Fixed Layout

Width is a fixed pixel value. Wide screens leave empty space; narrow screens require horizontal scrolling.

💧 Fluid Layout

Width is a percentage. Layout fills available space at any screen size.

Building a Fluid Layout

Container with Percentage Children

/* Container takes full width up to a max */
#container {
    width: 100%;
    max-width: 1200px;
    margin: 0 auto;
}

/* Two-column layout using percentages */
.column {
    width: 50%;
    float: left;
    padding: 1rem;
}

🔍 Media Queries

A media query is a CSS feature that detects the media type and capabilities of the device viewing the page, then applies different style rules accordingly. Most often, media queries detect viewport width and switch the layout when the screen passes a specific size, called a breakpoint.

The Shape of a Media Query

A media query starts with the @media keyword, followed by a condition in parentheses, then a block of CSS rules in curly braces. The rules inside only apply when the condition is true.

A Media Query for Tablets and Larger

/* Base styles apply to mobile (and everything) */
.column {
    width: 100%;
}

/* When viewport is 768px or wider, use two columns */
@media (min-width: 768px) {
    .column {
        width: 50%;
        float: left;
    }
}

/* When viewport is 1200px or wider, use three columns */
@media (min-width: 1200px) {
    .column {
        width: 33.33%;
    }
}

Common Breakpoints

There is no official list of breakpoints, but these widths are widely used because they correspond to common device sizes.

Breakpoint Targets
480px Larger phones in landscape
768px Tablets in portrait
1024px Tablets in landscape and small laptops
1200px Standard desktops
Mobile-first uses min-width. Start with mobile styles outside any media query. Then add @media (min-width: ...) blocks to enhance larger screens. The opposite approach (starting desktop and using max-width to shrink down) tends to produce slower, harder-to-maintain stylesheets.

🖼️ Responsive Images

Responsive images grow and shrink with their container so they look right at any screen size. The CSS that makes this work is short and applies globally.

Making All Images Responsive

img {
    max-width: 100%;
    height: auto;
}

max-width: 100% means the image never exceeds the width of its container. height: auto preserves the aspect ratio so the image doesn't get stretched or squashed.

Strip the inline width and height attributes. If your <img> tags still have width="800" and height="450" attributes from a previous design, those can fight your CSS rules. Remove them and let CSS handle sizing.

📌 Sticky Elements

A sticky element behaves like a normal block element until you scroll past where it would normally sit, at which point it sticks to a chosen edge of the viewport and stays visible. Sticky headers and navigation bars are the most common use.

📌 This nav bar stays at the top as you scroll

Scroll this box to see the sticky bar in action. The bar will pin to the top of this container as you scroll.

Sticky positioning is perfect for site navigation, table headers, and any UI element that should remain accessible while the user reads content below it.

Unlike fixed positioning, a sticky element only sticks within its parent. Once you scroll past its parent container, the sticky element scrolls away too.

This makes sticky positioning ideal for section headers in a long article, where each header should pin while you read its section, then release when you reach the next one.

The browser handles all the scroll logic automatically. Just set position: sticky and a top value, and the element does the rest.

Creating a Sticky Header

Sticky CSS

header {
    position: sticky;
    top: 0;
    background-color: white;
    z-index: 100;
}
  • position: sticky turns on the sticky behavior.
  • top: 0 tells the element which edge to stick to. With top: 0, it sticks to the top of the viewport. top: 20px would leave a 20-pixel gap.
  • z-index ensures the sticky element appears above other content as it sticks.

🎨 Custom Fonts

The default fonts available on every device are limited. Custom fonts let you use any typeface, regardless of whether the visitor has it installed locally. Two methods are common.

Method 1: Linking from Google Fonts

Google Fonts is a free library of hundreds of typefaces. To use one, add a single <link> in the HTML head, then reference the font in your CSS.

Linking Google Fonts

<!-- In the HTML head -->
<link href="https://fonts.googleapis.com/css2?family=Roboto&display=swap"
      rel="stylesheet">

/* In the CSS */
body {
    font-family: 'Roboto', sans-serif;
}

Method 2: Self-Hosted Custom Fonts

Custom fonts can also be downloaded onto your own web server. Use the @font-face rule in your CSS to define the typeface, then reference it like any other font.

Defining a Self-Hosted Font

@font-face {
    font-family: 'BrandFont';
    src: url('fonts/brandfont.woff2') format('woff2');
}

h1 {
    font-family: 'BrandFont', Georgia, serif;
}
Always include a fallback. Custom fonts can fail to load on slow connections. End the font-family with a system font and a generic family so the page still has reasonable typography even if the custom font never arrives.

🎯 Pseudo-Classes

A pseudo-class is a special selector that targets elements based on their state or position rather than their tag name, ID, or class. Pseudo-classes are written with a colon (:) followed by the state name.

Common Pseudo-Classes

a:link Unvisited Link
a:visited Visited Link
a:hover Mouse Hovering
a:active Being Clicked
a:focus Keyboard Focused
li:first-child First List Item
Pseudo-Class Matches
:link An unvisited link
:visited A link the visitor has clicked before
:hover An element the mouse pointer is over
:active An element being clicked or tapped right now
:focus An element that has keyboard focus
:first-child An element that is the first child of its parent
:last-child An element that is the last child of its parent
:nth-child(n) The nth child of its parent

Styling Link States

a:link { color: #475569; }
a:visited { color: #581C87; }
a:hover { color: #06B6D4; }
a:active { color: #BE123C; }
Order matters for link states. Browsers apply link pseudo-classes in this order: link, visited, hover, active. Writing them out of order can cause some rules to silently override others. Remember the mnemonic LVHA: Link, Visited, Hover, Active.

🧭 Mobile-Friendly Navigation

A navigation system designed for a desktop won't work well on a phone. Horizontal menus with many items become cramped and tap targets become too small. Mobile navigation usually adapts in two stages.

📱 On Mobile

Stack navigation links vertically in a single column. Each link gets generous padding so it's easy to tap. Some sites collapse the entire menu behind a "hamburger" button to save space.

💻 On Desktop

Use a media query to switch the same nav element to a horizontal row when the viewport is wide enough. The same HTML serves both layouts.

A Responsive Nav Pattern

/* Mobile: stacked vertical menu */
nav ul {
    list-style: none;
    padding: 0;
}
nav li {
    border-bottom: 1px solid #CBD5E1;
}
nav a {
    display: block;
    padding: 1rem;
    text-decoration: none;
}

/* Desktop: horizontal row */
@media (min-width: 768px) {
    nav li {
        display: inline-block;
        border: none;
    }
}

✨ Mobile-Friendly Touches

Telephone Links

Telephone links use tel: in the href. On a mobile device, tapping them starts a phone call. They are especially useful in mobile navigation, contact pages, and footer information.

A Tappable Phone Number

<a href="tel:+15555551234" class="tel-link">
    Call (555) 555-1234
</a>

Rounded Corners

The border-radius property gives any element rounded corners. Modern mobile design often uses generous corner radii on cards, buttons, and images for a softer, friendlier feel.

Rounded Corner Examples

/* Slightly rounded button */
.button {
    border-radius: 8px;
}

/* Pill-shaped tag */
.tag {
    border-radius: 999px;
}

/* Circular avatar */
.avatar {
    border-radius: 50%;
}

🧪 Testing on Mobile

Browser Device Mode

Modern browsers include built-in tools to simulate mobile devices. In Chrome, Edge, and Firefox, open the developer tools (F12 or right-click and choose "Inspect"), then toggle the device toolbar. The page will render as if it were on a phone or tablet, and you can switch between common device sizes from a menu.

Mobile Simulators

A mobile simulator shows what a website will look like on a particular phone or tablet. Browser device mode is the fastest option, but standalone simulators provided by Apple, Google, and others give a more accurate preview because they actually run the device's operating system.

Mobile-Friendly Tests

Search engines provide free tests that grade a published webpage on its mobile-friendliness. They check things like text size, tap target spacing, and viewport configuration, and report any problems with specific recommendations to fix them.

Test on real devices. Simulators are convenient, but they can't replicate every real-world condition. Test on actual phones whenever possible, especially the kind of phones your audience is most likely to use.

✨ Best Practices

Habits for mobile-first responsive design

  • Add the meta viewport element to every HTML file before doing anything else.
  • Design mobile first. Write base CSS for the smallest screen, then enhance with min-width media queries.
  • Use relative units. Prefer %, rem, and em over fixed pixels for sizing.
  • Make tap targets at least 44 by 44 pixels. Smaller buttons are hard to tap accurately.
  • Use a single column on mobile. Avoid horizontal scrolling at all costs.
  • Make every image responsive with max-width: 100%; height: auto;.
  • Choose readable font sizes. 16px or larger for body text.
  • Order LVHA when styling link states.
  • Provide font fallbacks in case custom fonts fail to load.
  • Test on real devices, not just simulators.
