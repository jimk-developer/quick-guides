🌉 From Mobile to Tablet and Desktop

Mobile-first design starts simple: a single-column layout, generous tap targets, and content that fits a phone screen without horizontal scrolling. Once that foundation is solid, the next job is to enhance the layout for visitors on bigger screens.

Tablets and desktops give you something phones can't: horizontal space. With more room to work, you can put navigation links in a row instead of a column, place content side by side, and use larger images. The technique that makes all of this possible is the media query.

Mobile styles flow upward. CSS rules written outside any media query apply to every screen size, including tablets and desktops. Media queries layer on additional rules only when the viewport is larger. You never need to undo mobile styles for bigger screens; you just override or extend them.

🔍 Media Queries in Depth

A media query is a CSS expression that detects characteristics of the device and applies different styles when those characteristics match. Most often you will use them to test the viewport's width, but they can test much more.

The @media Rule

Every media query starts with the @media rule (also called the @media at-rule). It is followed by an optional media type, an optional feature expression, and then a block of CSS rules in curly braces.

@media only screen and (min-width: 768px) { ... }
@mediaThe keyword that begins every media query
Media typescreen, print, all, or speech
Featurethe characteristic being tested
Valuethe threshold being compared

Media Types

A media type identifies the broad category of device. Most modern queries use screen or omit the type entirely.

Media Type Targets
all All devices. The default if no type is specified.
screen Phones, tablets, laptops, desktops, anything with a screen.
print Documents being printed or shown in print preview.
speech Screen readers and speech synthesizers.

The "only" Keyword

Adding the keyword only before the media type tells very old browsers to ignore the entire media query. Older browsers that don't understand the syntax skip it instead of trying to parse it incorrectly.

Using "only" for Backward Compatibility

@media only screen and (min-width: 768px) {
    /* Tablet and larger styles */
}

Common Media Query Features

Feature Tests Example
min-width Viewport at least this wide (min-width: 768px)
max-width Viewport no wider than this (max-width: 480px)
min-height Viewport at least this tall (min-height: 600px)
orientation Portrait or landscape (orientation: landscape)
resolution Screen pixel density (min-resolution: 2dppx)
prefers-color-scheme User's light or dark mode preference (prefers-color-scheme: dark)

Where Media Queries Go

Media queries can live in two places. Both work; they just organize the code differently.

📄 Inside the External Stylesheet

Most common approach. All responsive rules live in the same CSS file as the rest of your styles. One file to maintain, one file to validate.

🔗 On the link Element in HTML

The HTML <link> tag accepts a media attribute. The browser only loads the stylesheet when the media query matches. Useful for very large stylesheets specific to one device.

Two Ways to Apply a Media Query

<!-- Method 1: media attribute on the link element -->
<link rel="stylesheet"
      href="desktop.css"
      media="only screen and (min-width: 1024px)">

/* Method 2: @media rule inside an external stylesheet */
@media only screen and (min-width: 1024px) {
    .column { width: 33%; }
}

📱 Designing for Tablet Viewports

Tablets sit between phones and desktops. Most have screens between 7 and 13 inches. The same layout rarely fits both a phone and a tablet, but it can with a single media query.

Tablet Design Principles

↔️ Horizontal Navigation

The vertical stack of links from mobile becomes a row of links across the top. There is enough room to lay them out side by side without crowding.

📏 Side-by-Side Content

Sections that stacked vertically on mobile can sit next to each other on a tablet, often in a two-column arrangement.

🎨 Same Look and Feel

Color scheme, typography, and brand should not change between viewport sizes. Only the layout changes. Visitors should recognize your site instantly on any device.

📐 Generous Tap Targets

Tablets are still touchscreens. Buttons and navigation links need to stay finger-sized, even with the extra horizontal space.

A Tablet Media Query

The conventional breakpoint for tablets is around 768 pixels, the typical width of a tablet in portrait orientation. Use min-width: 768px to apply tablet styles to anything that wide or wider.

Switching to a Horizontal Nav at Tablet Width

/* Mobile base styles: vertical nav */
nav li {
    display: block;
    border-bottom: 1px solid #ccc;
}

/* Tablet and up: horizontal nav */
@media only screen and (min-width: 768px) {
    nav li {
        display: inline-block;
        border: none;
    }
}

Hiding and Showing Content per Viewport

The CSS display property gives you fine control over which elements appear at which sizes. Two utility classes that show up in nearly every responsive site:

Mobile-Only and Tablet-Desktop-Only Classes

/* Default: show .mobile, hide .tablet-desktop */
.tablet-desktop { display: none; }

/* At tablet+: hide .mobile, show .tablet-desktop */
@media only screen and (min-width: 768px) {
    .mobile { display: none; }
    .tablet-desktop { display: block; }
}

Now any element with class mobile appears only on phones, and any element with class tablet-desktop appears only on bigger screens.

🖥️ Designing for Desktop Viewports

Desktop screens are wider than every other category. Modern monitors range from 1280 pixels to 4K and beyond. The challenge isn't fitting content into a desktop. The challenge is using all that space without making the page feel empty or making text painfully wide to read.

Desktop Design Principles

📏 Constrain Content Width

Set a max-width on the main container. Stretching text the full width of a 27-inch monitor makes lines so long they become hard to follow. Common values are 1200 to 1400 pixels.

📰 Multi-Column Layouts

Desktop is the right place for three or more columns of content. The page can show a sidebar, primary content, and a related-content panel all at once.

🎯 Hover Interactions

Desktops have mice. Pseudo-classes like :hover can reveal extra information, animate buttons, and provide cues that touch devices can't show.

🖼️ Larger Imagery

Hero images, background photos, and feature graphics can fill more space and create visual impact that smaller screens can't match.

A Desktop Media Query

The standard desktop breakpoint is around 1024 pixels. Below that you treat the device as a tablet; above it, you apply desktop layout.

Centering Content with a Max Width

@media only screen and (min-width: 1024px) {
    #container {
        max-width: 1200px;
        margin: 0 auto;
    }

    .column {
        width: 33.33%;
        float: left;
    }
}

Large Desktop Viewports

Some sites add a third media query for very large screens (1600 pixels and up). At that size you might increase the container width, scale up typography, or show an extra sidebar that wasn't visible at the standard desktop size.

Don't use 100% of a wide screen for text. Studies suggest the most readable line length for body text is between 50 and 75 characters per line. On a 27-inch monitor at full width, paragraphs sail well past 200 characters per line. A max-width on the content container is essential for readability.

🖨️ Print Media Queries

Media queries can also produce printer-friendly versions of a webpage. A print stylesheet hides screen-only elements (like navigation menus) and adjusts colors for paper.

Common Print Adjustments

  • Hide navigation, sidebars, and footers. Visitors who print a page usually want the content, not the menus.
  • Remove background colors and images. Printer ink is expensive, and dark backgrounds with light text don't print well.
  • Use black text on white. Maximum contrast and readability on paper.
  • Use point sizes. Print uses physical units (pt, in, cm) rather than pixels.
  • Show full URLs after links. Readers can't click on paper, so they need to see where a link would go.

A Basic Print Stylesheet

@media print {
    /* Hide screen-only elements */
    nav, footer, .ads {
        display: none;
    }

    /* Use black on white */
    body {
        color: black;
        background: white;
        font-size: 12pt;
    }

    /* Show URLs after every link */
    a::after {
        content: " (" attr(href) ")";
    }
}

📏 Modifying Breakpoints

A breakpoint is the viewport width at which the layout changes. Standard values like 768 and 1024 work well as starting points, but the real answer is to set breakpoints based on your content, not specific device sizes.

Why Standard Breakpoints Aren't Always Right

Imagine your navigation has six links and they fit nicely on one line at 800 pixels but wrap awkwardly to a second line at 768 pixels. Sticking with 768 leaves your nav looking broken between 768 and 800 pixels. Adjust the breakpoint up to 800 and the problem disappears.

The right way to find breakpoints: resize your browser window slowly from narrow to wide. Watch for any layout that looks broken, cramped, or wasteful. That moment is where a breakpoint belongs.

Browser Developer Tools

Every modern browser includes built-in developer tools that make finding breakpoints easy.

📐 Resize Live

Open the page and drag the browser window narrower or wider. Watch the layout adjust at each breakpoint and look for visual problems.

📱 Device Mode

Toggle the device toolbar (often shaped like a phone icon) to simulate specific phones, tablets, and desktops at exact pixel sizes.

🎯 Custom Width

Type any specific viewport width into the device mode. Useful for testing the exact pixel value where your layout breaks.

🔍 Inspect Elements

Right-click any element and choose Inspect to see exactly which CSS rules are applied at the current viewport size.

Common shortcuts: press F12 (or Ctrl+Shift+I on Windows, Cmd+Option+I on Mac) to open developer tools. Press Ctrl+Shift+M (or Cmd+Shift+M on Mac) to toggle device mode in Chrome and Edge.

🎯 Dynamic Pseudo-Classes

A dynamic pseudo-class styles an element based on something the user is doing right now: pointing at a link, clicking it, or having visited it earlier. Dynamic pseudo-classes give you fine control over how interactive elements respond.

The Four Link Pseudo-Classes

Pseudo-Class Triggers When Typical Use
a:link The link has not been visited Default link color
a:visited The link has been visited (cached in browser history) Indicates content already seen
a:hover The mouse pointer is over the link Visual feedback that the link is interactive
a:active The link is currently being pressed Brief feedback during the click

Try It: Hover and Click

Hover Over Me Then Click Me

The buttons change appearance on hover and again on click. That's :hover and :active at work.

Construction of a Pseudo-Class

A pseudo-class attaches to a regular selector with a colon and the state name. There is no space between the selector and the colon.

Pseudo-Class Syntax

selector:state { /* style rules */ }

/* Real examples */
a:hover { color: crimson; }
nav a:hover { background-color: #fef2f2; }
.button:active { transform: translateY(1px); }

The Required Order

Browsers apply link pseudo-classes by cascade order: the last matching rule wins. If you put them out of order, some rules silently override others.

The LVHA Order

Always declare link pseudo-classes in this exact order in your stylesheet:

  1. a:link
  2. a:visited
  3. a:hover
  4. a:active

Mnemonic: Lord Vader Hates Anakin. Skip the order and your hover style might not work because :link declared later overwrites it.

Mobile Touch Caveat

Mobile devices don't have hover. A finger isn't a mouse pointer. There's no "hovering" state on a touchscreen. Some touch browsers simulate :hover after the first tap, but the behavior is inconsistent. Treat :hover as enhancement for mouse users, never as the only way to access important content.

🌈 Gradients

A gradient is a smooth transition from one color to another. CSS supports two kinds of gradients: linear (color flows in a straight line) and radial (color flows outward from a center point). Both are generated by the browser, with no images required.

Linear Gradients

A linear gradient transitions color along a straight line. By default, the line runs top to bottom, but you can specify any angle.

to bottom (default)
linear-gradient(crimson, pink)
to right
linear-gradient(to right, crimson, pink)
45 degrees
linear-gradient(45deg, ...)
three colors
linear-gradient(to right, ...)

Linear Gradient Examples

/* Default direction: top to bottom */
.banner {
    background: linear-gradient(crimson, pink);
}

/* Specify direction with "to right" or "to bottom right" */
.hero {
    background: linear-gradient(to right, #BE123C, #F472B6);
}

/* Specify direction with an angle in degrees */
.card {
    background: linear-gradient(45deg, #BE123C, #F472B6);
}

/* More than two colors */
.rainbow {
    background: linear-gradient(to right, red, yellow, green, blue);
}

Linear Gradient Direction Values

Value Direction of Flow
(omitted) Top to bottom (default)
to top Bottom to top
to right Left to right
to left Right to left
to bottom right Top-left corner to bottom-right corner
0deg to 360deg Any specific angle (0deg points up, 90deg points right)

Radial Gradients

A radial gradient transitions color from a center point outward in a circle or ellipse. The first color appears at the center and the last color appears at the edge.

default ellipse
radial-gradient(crimson, pink)
circle
radial-gradient(circle, ...)
offset center
radial-gradient(circle at top right, ...)

Radial Gradient Examples

/* Default: ellipse, centered */
.spotlight {
    background: radial-gradient(crimson, pink);
}

/* Force a circle instead of an ellipse */
.orb {
    background: radial-gradient(circle, #FBBF24, #BE123C);
}

/* Move the center using "at" */
.glow {
    background: radial-gradient(circle at top right, white, gray);
}
Two-color minimum. Both linear and radial gradients require at least two colors. With one color, there's nothing to transition between.

When to Use Gradients

🎨 Hero Backgrounds

A subtle gradient on a hero section adds depth without the file size of an image.

🎯 Buttons and Cards

A gentle gradient on a button or card adds dimensionality and makes the element feel more clickable.

🖼️ Image Overlays

A gradient layered over an image (using linear-gradient with transparency) makes overlaid text more readable.

🎉 Brand Accents

A gradient using your brand colors makes a memorable visual signature.

✨ Best Practices

Habits for tablet and desktop responsive design

  • Build mobile first, then layer up. Add media queries with min-width rather than starting at desktop and shrinking down.
  • Set breakpoints based on your content. Resize the browser slowly and add a breakpoint wherever the layout breaks.
  • Cap content width on desktop. Use a max-width on the container so lines don't get unreadably long.
  • Maintain visual consistency. Colors, fonts, and brand identity stay the same across viewports. Only layout changes.
  • Order link pseudo-classes LVHA. Out of order, some rules will silently fail.
  • Use :hover as enhancement, not as a hard requirement. Touch devices don't hover.
  • Add a print stylesheet for content-heavy pages. Hide nav, use black text, show URLs after links.
  • Use gradients sparingly. A subtle gradient adds polish; a loud gradient looks dated.
  • Test with browser dev tools at every breakpoint. Check mobile, tablet portrait, tablet landscape, desktop, and large desktop.
