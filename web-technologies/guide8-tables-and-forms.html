📊 Discovering Tables

An HTML table presents information in rows and columns. Tables are the right tool whenever you have data that compares well in a grid: a class schedule, a price list, a nutrition label, a sports standings table, a product feature comparison.

Table Anatomy

Every table is made of the same building blocks. Knowing the names is essential because each one has its own HTML element.

Core Table Elements

Element Purpose
<table> Wraps the whole table.
<caption> The title of the table. Goes immediately after the opening table tag. Centered by default.
<thead> Groups the header rows of the table.
<tbody> Groups the body rows of the table.
<tfoot> Groups the footer rows (such as totals).
<tr> Defines a single table row.
<th> A header cell. Bold and centered by default. Identifies a row or column.
<td> A data cell. Holds an actual value.
th vs. thead. These look similar but mean different things. <th> is a single header cell. <thead> is a wrapper around an entire group of header rows. A table has many <th> elements, but at most one <thead>.

Building a Simple Table

<table>
    <caption>Bakery Hours</caption>
    <thead>

        </tr>
    </thead>
    <tbody>
        <tr>
            <td>Tuesday</td>
            <td>7:00 AM</td>
            <td>2:00 PM</td>
        </tr>
        <tr>
            <td>Wednesday</td>
            <td>7:00 AM</td>

        </tr>

</table>

The 1990s Layout Misuse

In the late 1990s, before CSS was widely supported, web designers used tables to lay out entire pages: headers, sidebars, columns, and footers were all forced into table rows and cells. The HTML was bloated, inaccessible to screen readers, and a nightmare to maintain.

Never use tables for layout. Tables are for tabular data. Period. For page layouts, use CSS Grid or Flexbox. Misusing tables hurts accessibility (screen readers announce them as tables) and search engine optimization (search engines expect tabular content inside tables).

When to Use a Table

✅ Good Table Uses

  • Class schedules
  • Price lists
  • Nutrition information
  • Sports standings
  • Comparison charts
  • Financial data

❌ Wrong Tool

  • Page layout
  • Image galleries
  • Card grids
  • Navigation menus
  • Form layouts
  • Multi-column text

Planning a Table

Sketch Before You Code

  1. Identify the data. What rows and columns are essential? What can you cut?
  2. Decide what's a header. Header cells go in the top row, in the leftmost column, or both.
  3. Plan for mobile. Will this table fit on a phone? If not, plan an alternative layout for narrow screens before you start coding.
  4. Add a caption. Even short tables benefit from a clear title.

🎨 Styling Tables

Default browser styles for tables are spartan. A few CSS properties transform a plain table into something readable and attractive.

Common Table CSS Properties

Property Effect
border Adds borders to the table, rows, or cells.
border-collapse Combines adjacent cell borders into a single line, or keeps them separate.
border-spacing Distance between cells when borders are separated.
padding Space between cell content and cell edges. Apply to th and td.
text-align Horizontal alignment of cell content (left, center, right).
vertical-align Vertical alignment within a cell (top, middle, bottom).
background-color Background color, often used for header rows or alternating data rows.

Border Styles: Separated vs. Collapsed

The border-collapse property has two main values that produce very different looks.

A Polished Table Style

table {
    border-collapse: collapse;
    width: 100%;
}

th, td {
    padding: 12px 16px;
    text-align: left;
    border-bottom: 1px solid #ddd;
}

th {
    background-color: #B45309;
    color: white;
}

/* Striped rows for easier scanning */
tbody tr:nth-of-type(even) {
    background-color: #FFFBEB;
}

/* Highlight on hover */
tbody tr:hover {
    background-color: #FEF3C7;
}

Responsive Tables

Tables present a particular challenge on phones. A table with five or six columns simply won't fit on a 320-pixel screen without horizontal scrolling. Several patterns address this.

📜 Horizontal Scroll

Wrap the table in a div with overflow-x: auto. Visitors can swipe sideways to see all columns. Simplest solution; works for most data.

🔄 Stacked Cards

On mobile, display each row as a vertical card with labels next to each value. Each "row" becomes its own block.

🎯 Hide Columns

Use CSS to hide the least important columns on small screens, keeping only the essential ones visible.

📊 Reformat as Chart

Long numerical tables are sometimes more useful as a chart on mobile. Show the chart on phones and the full table on tablets and desktops.

The Stacked-Card Pattern

Here's how a four-column table might display when transformed into stacked cards on mobile:

The Simplest Responsive Table: Horizontal Scroll

<!-- Wrap the table in a scrolling container -->
<div class="table-scroll">
    <table> ... </table>
</div>

/* CSS */
.table-scroll {
    overflow-x: auto;
}

📝 Creating Webpage Forms

A form is a section of a webpage that collects information from the visitor. Contact forms, search boxes, login screens, sign-up flows, surveys, and checkout pages all use HTML forms. Whenever a visitor types something into a website, they're filling out a form.

What Forms Collect

📇 Contact Information

Name, email, phone, mailing address.

🔐 Account Credentials

Username, password, security questions.

🛒 Purchases and Orders

Cart items, shipping info, payment details.

📊 Feedback and Surveys

Ratings, opinions, multiple-choice answers, open comments.

The form Element

Every form starts with a <form> element. This element wraps all the controls (input fields, buttons, etc.) and tells the browser what to do when the form is submitted.

The Skeleton of a Form

<form action="/contact-process" method="post">
    <!-- input controls go here -->

    <button type="submit">Send</button>
</form>

Common Form Attributes

Attribute Purpose
action The URL where the browser sends form data when submitted. The server-side script that receives the data.
method How the browser sends the data: get or post.
name Identifies this form when there's more than one on a page.
autocomplete Whether the browser should suggest values from past form submissions.

The get Method vs. the post Method

📨 method="get"

Form data appears in the URL as query parameters.

/search?term=sourdough&category=bread

Use for: searches, filters, and any data the visitor would want to bookmark or share.

📤 method="post"

Form data is sent in the body of the HTTP request, hidden from the URL.

/contact-process

Use for: form submissions that change something on the server (sign-ups, orders, contact messages).

Never use get for sensitive data. Passwords, credit card numbers, and personal information should always go through post. URLs are saved in browser history, server logs, and HTTP referrer headers.

🎛️ Input Controls

The input element is the workhorse of HTML forms. A single <input> tag can be a text box, a checkbox, a radio button, a date picker, or a submit button, depending on the value of its type attribute.

Common Input Attributes

Attribute Purpose
type Determines what kind of input control to render.
name The key under which the value is sent to the server. Required for the value to be included in the submission.
id Unique identifier for the control. Used to associate a label with the input.
value The starting value of the control, or for buttons, the text on the button.
placeholder Hint text shown in an empty text field. Disappears when the visitor starts typing.
maxlength Maximum number of characters the user can enter.
size Visible width of a text input in characters.
required The form won't submit unless this control has a value.

The Input Types

The type attribute changes everything about an input. Here are the most common values, each shown in action.

type="text"

A single-line text box. The default if you don't specify a type.

type="email"

Like text, but validates that the value looks like an email address. Mobile keyboards show an @ key.

type="tel"

Phone number input. Mobile keyboards show a numeric keypad.

type="password"

Masks the typed characters as dots or asterisks. Doesn't add encryption; only hides the display.

type="date"

A date picker. Browsers display a calendar widget. The value is always in YYYY-MM-DD format regardless of locale.

type="number"

A numeric input with up/down spinners. Use min and max attributes to restrict the range.

type="checkbox"

An on/off toggle. Multiple checkboxes can be selected at once.

type="radio"

One-of-many selector. Radio buttons with the same name form a group; only one can be selected.

type="submit"

The Submit button. Sends the form to the server when clicked.

type="reset"

The Reset button. Clears all controls back to their initial values.

Radio buttons must share a name. Radio buttons only behave as a single-choice group when they all have the same name attribute. Different names mean separate groups, and the visitor can pick one from each.

The textarea Element

The <textarea> element creates a multi-line text input. Unlike <input>, textarea is a paired tag, with the default text between the opening and closing tags. The visible size is set with the rows and cols attributes.

A Textarea

<label for="comments">Comments</label>
<textarea id="comments"
          name="comments"
          rows="5"
          cols="40">
</textarea>

The select Element

The <select> element creates a dropdown menu. Each choice is an <option> element nested inside.

A Dropdown Menu

<label for="flavor">Flavor</label>
<select id="flavor" name="flavor">
    <option value="">Choose one</option>
    <option value="vanilla">Vanilla</option>
    <option value="chocolate">Chocolate</option>
    <option value="strawberry">Strawberry</option>
</select>
Pre-select with the selected attribute. Add selected to whichever <option> should be the default choice when the form first appears. If you don't, the first option is selected.

🏷️ Labels: The Most Important Element

The label element attaches a text label to a form control. Labels matter for three reasons: clicking the label focuses the control (especially useful for small checkboxes), screen readers announce the label when the visitor reaches the control, and the form is much easier to use overall.

Two Ways to Associate a Label

🎯 Explicit (preferred)

Use a for attribute on the label that matches the id of the input.

<label for="email">Email</label>
<input type="email" id="email">

🎁 Implicit

Wrap the input inside the label.

<label>
    Email
    <input type="email">
</label>
Always pair a label with every form control. Submit buttons and reset buttons are the only exception (they have their text on the button itself). Every other input, textarea, and select needs a label.

📋 Grouping with fieldset and legend

The fieldset element groups related form controls together. The legend element provides a caption for the group. Together they make complex forms easier to scan and understand.

Common groupings include "Contact Information", "Shipping Address", "Account Preferences", and so on. Visually, fieldset draws a thin border around the group, and legend appears as a label that breaks through the top border.

A Fieldset with Legend

<fieldset>
    <legend>Newsletter Preferences</legend>

    <label>
        <input type="checkbox" name="weekly">
        Weekly recipe roundup
    </label>

    <label>
        <input type="checkbox" name="events">
        Class and event announcements
    </label>
</fieldset>

💅 Styling Forms

Default browser form styles look dated. A small amount of CSS modernizes them and makes the form match the rest of your site.

Common Form Styling Targets

📐 Consistent Sizing

Set a standard padding and font-size on all inputs, textareas, and selects. Browser defaults vary widely; a uniform size makes the form feel intentional.

🎯 Focus States

Style the :focus state with a colored border or shadow so visitors can see clearly which control is active.

🔘 Button Styling

Submit buttons get the brand color. Reset buttons (when used) get a more neutral color so they don't compete with the primary action.

📏 Mobile Layout

Stack labels above inputs on phones. On tablets and desktops, you can place them side by side if there's room.

A Modern Form Style

input, textarea, select {
    width: 100%;
    padding: 10px 14px;
    border: 2px solid #ddd;
    border-radius: 6px;
    font-size: 1rem;
}

input:focus, textarea:focus, select:focus {
    outline: none;
    border-color: #D97706;
    box-shadow: 0 0 0 3px rgba(217,119,6,0.2);
}

button[type="submit"] {
    background: #D97706;
    color: white;
    padding: 12px 30px;
    border: none;
    border-radius: 6px;
    cursor: pointer;
    font-weight: 600;
}

✨ Best Practices

Habits for tables and forms

  • Use tables only for tabular data. Never for layout.
  • Add a caption to every table. Even short tables benefit from a clear title.
  • Mark up header cells with <th>. Screen readers depend on this to navigate the data.
  • Pair a label with every form control. Required for accessibility and for usability.
  • Use the right input type. Email, tel, date, and number give visitors better keyboards on mobile and built-in validation.
  • Group related inputs with fieldset and legend. Especially in forms with more than five fields.
  • Use post for sensitive data. Never put passwords or personal info in a URL with get.
  • Style focus states. Visitors should always see clearly which input has keyboard focus.
  • Plan tables for mobile from the start. Decide upfront whether to scroll, stack, hide, or chart.
  • Validate every page after adding forms. Forms introduce many new attributes and unique IDs that need checking.
