🗺️ From Plan to Template

Before writing any markup, a web designer takes a few planning steps. This keeps the website consistent across every page and prevents surprises later.

The Template Workflow

  1. Articulate the purpose of the website and identify the target audience.
  2. Create a site map showing how pages relate to each other.
  3. Design a wireframe showing the layout of a single page.
  4. Pick colors, fonts, and graphics that support the brand.
  5. Build a template containing the shared structure of every page.
  6. Clone the template to create each individual page.

Site Map vs. Wireframe

These two tools serve different purposes and are often confused. A site map is about the collection of pages. A wireframe is about the layout of one page.

🗂️ Site Map

Scope: Whole website

Shows: How many pages exist and how they link to each other

Answers: "How does someone get from the home page to Contact Us?"

📐 Wireframe

Scope: A single page

Shows: Where the header, navigation, main content, and footer sit on the page

Answers: "What goes where on this page?"

📁 File Management

Every website has many files: HTML documents, images, stylesheets, scripts, and more. Keeping these organized from day one prevents broken links and confusion later.

Root Folder and Subfolders

The root folder is the main folder that holds the entire website. Inside the root folder, you typically create subfolders to group related files. A common layout looks like this:

📁 my-website/ // the root folder ├── 📄 index.html // the home page ├── 📄 about.html ├── 📄 contact.html ├── 📁 css/ // stylesheets │ └── 📄 styles.css ├── 📁 images/ // photos, icons, logos │ ├── 🖼️ logo.png │ └── 🖼️ hero.jpg └── 📁 scripts/ // JavaScript files └── 📄 main.js
The home page belongs in the root. Name it index.html. Web servers automatically serve this file when someone visits your domain, which is why visitors don't have to type the filename in the address bar.

File Naming Rules

  • Use lowercase. Some web servers treat About.html and about.html as different files, so stay consistent.
  • No spaces in filenames. Use hyphens to separate words, such as contact-us.html.
  • Use descriptive names. product-list.html is clearer than page3.html.
  • Keep names short. Long filenames make URLs ugly and hard to share.

🧱 HTML 5 Semantic Elements

A semantic element is an HTML tag whose name describes the meaning of its content, not just its appearance. Semantic elements are also called structural elements because they define the structure of a webpage.

Why Semantic Elements Matter

♿ Accessibility

Screen readers use semantic tags to help visually impaired users navigate directly to the main content, skipping past menus and headers.

🔍 Search Engines

Search engines give more weight to content inside main and article than content in headers or footers.

🧹 Readability

Code is easier to scan when tag names describe their purpose. Another developer can understand your page at a glance.

🎨 Styling

Semantic tags give CSS and JavaScript meaningful hooks without requiring extra class names.

The div Element

Before HTML 5, web designers used the generic div element to divide a page into sections. The div element has no meaning of its own. It is just a container. Each div typically needed a unique name (given by an id or class attribute) to distinguish it from other divs.

HTML 5 added semantic elements so that common page regions have descriptive tag names. Many professional designers prefer these semantic tags over generic div elements whenever possible. You will still see div in modern code, but it is typically reserved for grouping content that doesn't fit any semantic tag.

Common Semantic Elements

Element Meaning Typical Content
<header> Identifies the top region of a page or section Site name, logo, tagline
<nav> Contains navigation links Menu of page links to other parts of the site
<main> The primary, unique content of the page Article text, product listing, form, or whatever the page is about
<section> A thematic grouping of content inside main Distinct topic areas, each with its own heading
<article> Self-contained content that could stand alone Blog post, news story, product card
<aside> Content related but tangential to the main content Sidebar, related links, pull quote, ads
<footer> Identifies the bottom region of a page or section Copyright, contact info, page links, social icons
Head vs. header vs. heading. These three words sound similar but mean different things.
  • <head> is the section of the document that holds meta information, such as the title, character encoding, and stylesheet links. The visitor never sees it directly.
  • <header> is a semantic element for the top of the visible page or section, such as a banner with your logo.
  • Heading refers to <h1> through <h6>, the six levels of titled text that organize content.

📐 Semantic Wireframes

A semantic wireframe is a wireframe that labels each region with the HTML 5 semantic element you plan to use. It is a planning tool that bridges the gap between design and code.

<header>
<nav>
<main>
<aside>
<footer>

Labeling the wireframe with semantic tags turns a visual sketch into a structural plan. When you sit down to write the HTML, the skeleton is already decided.

📄 Creating a Webpage Template

A template is a predefined HTML document containing the shared structure of every page in a website. Instead of rewriting the same header, navigation, and footer for each page, you write them once in the template and reuse it.

What Goes in a Template

  • The DOCTYPE declaration and the <html> wrapper
  • A complete <head> section (character encoding, viewport, title placeholder)
  • The semantic structure: <header>, <nav>, <main>, <footer>
  • Static content that appears on every page, such as the logo, navigation links, and copyright notice
  • Comments marking where each section ends and where unique page content will be added

A Basic Template Skeleton

Here is a starter template for a small business site. This is an original example to show the overall shape. Every element you have seen so far appears inside it.

<!DOCTYPE html>
<html lang="en">
<head>
    <meta charset="UTF-8">
    <meta name="viewport" content="width=device-width, initial-scale=1.0">
    <title>Prairie Bakery</title>
</head>
<body>

    <!-- Site header with logo -->
    <header>
        <h1>Prairie Bakery</h1>
    </header>

    <!-- Main site navigation -->
    <nav>
        Home  |  Menu  |  About  |  Contact
    </nav>

    <!-- Primary page content goes here -->
    <main>
        
    </main>

    <!-- Page footer with copyright -->
    <footer>
        &copy; 2026 Prairie Bakery
    </footer>

</body>
</html>
Save your work often. Web development involves small, frequent changes. Save after each edit so you never lose more than a minute of work if something goes wrong.

💬 HTML Comments

A comment is a note you leave inside an HTML document for yourself or other developers. Comments are ignored by the browser. They appear nowhere on the rendered page. They exist purely to explain the code.

Comment Syntax

The Shape of a Comment

<!-- This is a comment -->

Comments start with <!-- and end with -->. Anything between those markers is a comment, even if it spans multiple lines.

Why Use Comments

  • Label sections so you can find them quickly when scrolling through long files.
  • Explain decisions that aren't obvious from the code alone.
  • Leave reminders for yourself or teammates about work still to do.
  • Temporarily hide content or code during testing. The browser will skip over a commented-out block.

Good vs. Bad Comments

✅ Helpful

<!-- Featured product
     section. Updated
     weekly by marketing -->

Adds context that isn't obvious from the code.

❌ Unhelpful

<!-- header -->
<header>

Repeats what the tag already says. Adds noise without value.

Word Wrap in Your Editor

When an HTML line is long, it can run off the right edge of the editor window and become hard to read. Word wrap is a feature that causes text lines to break at the right edge of the window so the entire line stays visible. Word wrap only changes how the text looks in your editor. It does not change the file itself or affect the browser.

Blank lines in HTML are harmless. Browsers ignore blank lines when interpreting the page. You can add blank lines between sections of your HTML to make the source easier to read without changing how the page renders.

🔖 Heading Elements

HTML offers six levels of headings, from <h1> (most important) down to <h6> (least important). Headings organize content into a hierarchy and help both readers and search engines understand the structure.

<h1>Page Title
<h2>Major Section
<h3>Subsection
<h4>Smaller Heading
<h5>Even Smaller
<h6>Smallest Heading

Heading Rules of Thumb

  • Use one <h1> per page for the main page title.
  • Don't skip levels. Go from h1 to h2 to h3 in order. Jumping from h1 straight to h4 confuses screen readers.
  • Use headings for structure, not for size. If you just want bigger text, use CSS. Headings carry meaning, not just visual weight.
  • Keep heading text short and descriptive.

Static Content

Static content is the fixed text, images, and markup that doesn't change between visits to the page. A site logo, navigation menu, copyright notice, and page title are all static content. Static content is typically placed directly into a template so it appears on every page automatically.

🔣 Symbol Entities

Some characters cannot be typed directly into an HTML document. Either they have a special meaning in HTML (like the less-than sign, which starts a tag), or they are not on a standard keyboard (like the copyright symbol or an arrow).

To display these characters, HTML provides character entities. Every entity starts with an ampersand (&) and ends with a semicolon (;).

Two Ways to Write an Entity

🔤 Entity Name

A short memorable word between the ampersand and semicolon. Easier to read but not every character has a name.

Example: &copy; displays the copyright symbol.

🔢 Entity Number

A numeric code preceded by a hash sign. Works for every Unicode character, but harder to remember.

Example: &#169; also displays the copyright symbol.

Common Symbol Entities

Symbol Description Entity Name Entity Number
  Nonbreaking space &nbsp; &#160;
< Less-than sign &lt; &#60;
> Greater-than sign &gt; &#62;
& Ampersand &amp; &#38;
" Double quotation mark &quot; &#34;
© Copyright &copy; &#169;
® Registered trademark &reg; &#174;
™ Trademark &trade; &#8482;
€ Euro sign &euro; &#8364;
• Bullet &bull; &#8226;

The Nonbreaking Space

The nonbreaking space entity, written as &nbsp;, forces the browser to display a blank space. It is useful in two situations:

  • When you want extra space between items, because browsers collapse multiple regular spaces into one.
  • When you want to prevent a line break between two words, such as a first and last name or a number and its unit.

Using Nonbreaking Spaces in Navigation

<nav>
    Home &nbsp;&nbsp; Menu &nbsp;&nbsp; About &nbsp;&nbsp; Contact
</nav>

UTF-8 encoding (declared with <meta charset="UTF-8">) covers nearly every character in every language, which reduces the need for entity codes. You still need entities for HTML's special characters (<, >, &), but most symbols can simply be typed or copy-pasted into the document.

✅ Validating HTML Documents

Validation is the process of checking your HTML against the W3C standard. A validator reads your file, compares it to the official rules, and reports any mistakes it finds.

Why Validate

🐛 Catch Errors Early

Mistyped tags, unclosed elements, and missing attributes often fail silently in a browser. Validators surface them immediately.

🌐 Cross-Browser Consistency

Valid HTML renders more predictably across Chrome, Safari, Firefox, and Edge. Invalid HTML can behave differently in each.

♿ Accessibility

Assistive technology depends on valid markup. Errors in structure can make a page unusable for screen reader users.

🏆 Professional Quality

Validated pages are a hallmark of careful, professional work. They also rank better in search results.

The W3C Online Validator

The World Wide Web Consortium (W3C) provides a free online validator at validator.w3.org. You can use it three ways:

  • Validate by URI: paste the web address of a published page.
  • Validate by file upload: upload an HTML file from your computer.
  • Validate by direct input: paste HTML code directly into a text box.

The validator reports every issue it finds, along with the line number and a short explanation. Warnings are informational. Errors must be fixed for the document to be valid.

Common Validation Errors

Error What It Means How to Fix
Unclosed element A start tag has no matching end tag. Add the missing end tag.
Stray end tag An end tag has no matching start tag. Delete the stray end tag or add the missing start tag.
Missing required attribute An element is missing an attribute that is required by the HTML specification. Add the required attribute, such as alt on an image.
Duplicate ID Two elements share the same id value. Rename one of them. IDs must be unique on a page.
Element nested incorrectly A block-level element appears inside a tag where it isn't allowed. Restructure the markup so the nesting follows the rules.
Byte-order mark (BOM) warning. Some editors save files with an invisible byte-order mark at the start. The validator may report it as a warning. In most cases this is harmless, but if it causes trouble you can change your editor's save setting to "UTF-8 without BOM".

🏠 Creating Pages from a Template

Once you have a validated template, creating the rest of the site is quick. The process looks like this:

From Template to Home Page

  1. Open the template file in your editor.
  2. Save it with a new name using "Save As". For the home page, use index.html. Save it in the root folder.
  3. Update the title inside the <title> tag so the browser tab shows the name of this specific page.
  4. Add unique content inside the <main> element. This is what makes this page different from all the others.
  5. Save the file and open it in a browser to preview.
  6. Validate the new page using the W3C validator.

Adding Paragraphs to the Main Section

The <p> element defines a paragraph. Browsers automatically add space above and below each paragraph so blocks of text are visually separated.

Adding Content Inside main

<main>
    <h2>Welcome</h2>
    <p>We bake fresh bread and pastries every morning
       in small batches.</p>
    <p>Stop by any weekday between 7 AM and 2 PM.</p>
</main>
Uploading to a web server. To publish a site so the world can see it, you upload the files from your computer to a web server. Most hosting providers offer a web-based file manager or accept FTP. Once the files are on the server and the server is reachable through a domain name, your site is live.

✨ Best Practices

Habits that make template-based development smooth

  • Build the template first. Don't start coding individual pages until the template is finished and validated.
  • Use semantic tags whenever possible. Reach for <div> only when no semantic tag fits.
  • Comment your sections. A one-line comment at the top of each major region makes long files navigable.
  • One h1 per page. Use <h2> through <h6> for everything below it, in order.
  • Keep file and folder names lowercase with hyphens instead of spaces.
  • Validate before you publish. Catching errors in the W3C validator is faster than debugging them later.
  • Keep the template pristine. Never edit the master template when building a specific page. Save a copy under a new name first.
