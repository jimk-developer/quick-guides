Getting Started with React + TypeScript

Components, TSX, Typed Hooks, Forms, Data Fetching, and React 19 Features

🚀 Getting Started with React

React is a JavaScript library for building user interfaces. Instead of writing long, tangled HTML files and manipulating the DOM with vanilla JavaScript, React lets you break your UI into small, reusable pieces called components. Each component manages its own chunk of the screen, making your code easier to understand, test, and maintain.

In this guide, we use TypeScript from the ground up. TypeScript adds static type checking to JavaScript, catching bugs at compile time instead of at runtime. React files that contain TSX (the TypeScript version of JSX) use the .tsx extension. Files with plain TypeScript logic (no TSX markup) use .ts.

Why React + TypeScript?

React is used by millions of developers and powers apps at companies like Meta, Netflix, and Airbnb. Adding TypeScript gives you autocompletion in your editor, type safety for your props and state, and earlier error detection — your editor highlights mistakes before you even run the app. If you're building a MERN stack application with TypeScript on the backend, using TypeScript on the frontend means your entire stack speaks the same language.

Setup & Installation

The modern way to create a new React + TypeScript project is with Vite (pronounced "veet"). Vite is a lightning-fast build tool that replaces the older Create React App (CRA). Open your terminal and run:

Scaffolding a New React + TypeScript Project with Vite

# Create a new project with the React + TypeScript template
npm create vite@latest my-app -- --template react-ts

# Move into the project folder
cd my-app

# Install all dependencies
npm install

# Start the development server
npm run dev

Your app will launch at http://localhost:5173. Every time you save a file, the browser updates instantly — that's Vite's Hot Module Replacement (HMR) in action. Notice that Vite generates .tsx files instead of .jsx — that's the TypeScript template at work.

The Starting Point: createRoot

Every React app starts at one place: the main.tsx file. This file uses createRoot to "mount" your top-level component into the HTML page. This is the single React DOM API you need to kick off your app.

main.tsx — The Entry Point

import { StrictMode } from 'react';
import { createRoot } from 'react-dom/client';
import App from './App.tsx';
import './index.css';

// The "!" is a TypeScript non-null assertion — we know #root exists
createRoot(document.getElementById('root')!).render(
  <StrictMode>
    <App />
  </StrictMode>
);
StrictMode is a development-only wrapper. It doesn't render anything visible, but it helps catch potential problems in your code (like accidental side effects) by running components twice during development. The ! after getElementById('root') tells TypeScript: "I guarantee this element exists — it won't be null."

🧩 React Basics: Components & TSX

Thinking in Components

A component is a TypeScript function that returns a piece of UI. Every part of your page — the navbar, a product card, a footer — can be its own component. You build your app by composing these small pieces together, like LEGO bricks. Component files use the .tsx extension because they contain TSX markup.

<App />
↙ ↓ ↘
<Header />
<ProductList />
<Footer />
↙ ↘
<ProductCard />
<ProductCard />

A Simple Component (Greeting.tsx)

// Components are just functions that return TSX
function Greeting(): React.JSX.Element {
  return <h1>Hello, World!</h1>;
}

export default Greeting;
Return type annotation is optional. TypeScript can infer the return type of your components automatically. You'll see some codebases write function Greeting(): React.JSX.Element and others just write function Greeting() — both are fine. The explicit annotation can be helpful as documentation while you're learning.

Writing TSX

TSX is the TypeScript version of JSX. It looks like HTML, but it lives inside your .tsx files. Under the hood, React transforms TSX into regular JavaScript function calls. There are a few important rules:

TSX Rule Explanation
Return a single root element Wrap everything in one parent <div> or use a Fragment (<>...</>).
Close all tags Self-closing tags like <img> must be written as <img />.
Use className, not class Since class is a reserved word in JavaScript/TypeScript, TSX uses className.
Use htmlFor, not for Same reason — for is a reserved keyword. Labels use htmlFor.
JavaScript/TypeScript goes inside { } Use curly braces to embed variables, expressions, or function calls inside TSX. 
function UserCard() {
  const name: string = "Alice";
  const isOnline: boolean = true;

  return (
    <>                              {/* Fragment — no extra div in the DOM */}
      <h2 className="card-title">{name}</h2>       {/* Embed a variable */}
      <p>Status: {isOnline ? "🟢 Online" : "⚪ Offline"}</p>
      <img src="/avatar.png" alt="avatar" />   {/* Self-closing tag */}
    </>
  );
}

Using Props with TypeScript Interfaces

Props (short for "properties") are how you pass data from a parent component down to a child component. In TypeScript, you define an interface to describe the shape of the props object. This gives you autocompletion and type checking — if you misspell a prop name or pass the wrong type, TypeScript catches it immediately.

Defining and Using Typed Props

// Define an interface for the component's props
interface ProductCardProps {
  name: string;
  price: number;
  inStock?: boolean;   // "?" means this prop is optional
}

// Destructure props with the interface as the type annotation
function ProductCard({ name, price, inStock = true }: ProductCardProps) {
  return (
    <div className="card">
      <h3>{name}</h3>
      <p>${price.toFixed(2)}</p>
      <p>{inStock ? "In Stock" : "Out of Stock"}</p>
    </div>
  );
}
// Parent component passes data via attributes — TypeScript validates them
function App() {
  return (
    <div>
      <ProductCard name="Widget" price={9.99} />
      <ProductCard name="Gadget" price={24.99} inStock={false} />
      {/* ❌ TypeScript error: Property 'colour' does not exist */}
      {/* <ProductCard name="Gizmo" price={5} colour="red" /> */}
    </div>
  );
}
Props are read-only. A child component should never try to modify the props it receives. If you need data that changes over time, you need state (covered in the next section).

🪝 The Essential Hooks: State & Effects

Hooks are special functions that let your components "hook into" React features. The two most important hooks you'll use every day are useState and useEffect. In TypeScript, you can provide type parameters (generics) to hooks so that React and your editor know exactly what type of data you're working with.

useState — Giving Components Memory

useState lets you add a state variable to your component. State is data that can change over time. When the state changes, React automatically re-renders the component to show the updated UI.

Syntax: useState with TypeScript

import { useState } from 'react';

// Define a type for each item in the cart
interface CartItem {
  id: number;
  name: string;
}

function ShoppingCart() {
  // Provide the type parameter so TypeScript knows items is CartItem[]
  const [items, setItems] = useState<CartItem[]>([]);

  function addItem(): void {
    // Create a NEW array with the old items plus the new one
    setItems([...items, { id: Date.now(), name: "New Item" }]);
  }

  return (
    <div>
      <p>Items in cart: {items.length}</p>
      <button onClick={addItem}>Add Item</button>
    </div>
  );
}

How useState Works

1. useState<Type>(initialValue) returns an array with two things: the current value and a setter function.

2. You use array destructuring to give them names: [items, setItems].

3. Calling the setter (e.g., setItems(...)) tells React the state changed, and React re-renders the component with the new value.

4. Never modify state directly (e.g., items.push(...)). Always use the setter function with a new value.

When do you need the type parameter? For simple types like useState(0) or useState(""), TypeScript infers the type automatically and you can skip the generic. You need to provide it when the initial value doesn't tell the full story — like useState<CartItem[]>([]) (an empty array could be anything) or useState<User | null>(null) (it starts as null but will later hold a User).

useEffect — Running Code in the Background

useEffect lets you run "side effects" — code that reaches outside your component, like fetching data from your Express API, setting up a timer, or updating the document title.

Syntax: useEffect with TypeScript

import { useState, useEffect } from 'react';

interface User {
  _id: string;
  name: string;
  email: string;
}

interface UserProfileProps {
  userId: string;
}

function UserProfile({ userId }: UserProfileProps) {
  // Starts as null, will eventually hold a User object
  const [user, setUser] = useState<User | null>(null);

  // This runs AFTER the component renders
  useEffect(() => {
    // Fetch data from your Express API
    fetch(`/api/users/${userId}`)
      .then((res: Response) => res.json())
      .then((data: User) => setUser(data));
  }, [userId]);  // ← dependency array: re-run only when userId changes

  if (!user) return <p>Loading...</p>;

  return <h2>{user.name}</h2>;
}
Dependency Array When the Effect Runs
useEffect(() => { ... }) After every render (rarely what you want)
useEffect(() => { ... }, []) Only once, after the first render (like "on page load")
useEffect(() => { ... }, [userId]) After the first render, and whenever userId changes
The dependency array is the key. It tells React: "Only re-run this effect if one of these values has changed since the last render." Forgetting a dependency is one of the most common bugs in React — your linter will usually warn you about it.

🖱️ Handling User Interaction & Forms

Event Listeners

In React, you attach event handlers directly in your TSX using camelCase attributes like onClick, onChange, onMouseOver, and onKeyDown. You pass a function reference, not a function call. TypeScript provides specific event types like React.MouseEvent and React.ChangeEvent so you get full type checking on the event object.

import { useState } from 'react';

function LikeButton() {
  const [likes, setLikes] = useState<number>(0);

  // TypeScript event type for a button click
  function handleClick(e: React.MouseEvent<HTMLButtonElement>): void {
    setLikes(likes + 1);
  }

  return (
    <button onClick={handleClick}>
      👍 {likes} Likes
    </button>
  );
}
Pass the function, don't call it! Write onClick={handleClick}, NOT onClick={handleClick()}. Adding the parentheses would call the function immediately on every render, not when the user clicks.

Form Handling in React 19

React 19 introduced the ability to pass an action function directly to a form's action prop. This gives you a clean, modern way to handle form submissions without manually calling e.preventDefault(). The action function receives the native FormData object.

React 19 Form Actions with TypeScript

function ContactForm() {
  async function submitForm(formData: FormData): Promise<void> {
    const name: string = formData.get("name") as string;
    const email: string = formData.get("email") as string;

    await fetch("/api/contacts", {
      method: "POST",
      headers: { "Content-Type": "application/json" },
      body: JSON.stringify({ name, email }),
    });
  }

  return (
    <form action={submitForm}>
      <input name="name" placeholder="Name" />
      <input name="email" type="email" placeholder="Email" />
      <button type="submit">Send</button>
    </form>
  );
}
Why is this a big deal? In earlier versions of React, you had to wire up onSubmit, call e.preventDefault(), and manually read values from state or refs. With React 19 form actions, the form just works — React handles the submission and hands you a FormData object automatically. The as string cast is needed because formData.get() returns FormDataEntryValue | null.

Conditional Rendering & Lists

Two of the most common patterns you'll use in React are conditional rendering (showing or hiding UI based on state) and rendering lists (turning an array of data into an array of components using .map()).

interface Task {
  id: number;
  text: string;
  done: boolean;
}

function TaskList() {
  const [tasks, setTasks] = useState<Task[]>([
    { id: 1, text: "Learn React", done: true },
    { id: 2, text: "Build a project", done: false },
    { id: 3, text: "Deploy to the cloud", done: false },
  ]);

  return (
    <ul>
      {tasks.map((task: Task) => (
        <li key={task.id}>
          {/* Conditional rendering with the ternary operator */}
          {task.done ? "✅" : "⬜"} {task.text}
        </li>
      ))}
    </ul>
  );
}
Always include a key prop when rendering lists. React uses the key to efficiently track which items have changed, been added, or been removed. Use a unique, stable identifier (like an _id from your database) — never use the array index as the key if items can be reordered or deleted.

🔌 Connecting to Your Backend (e.g., Express)

Fetching Data

To connect your React front end to your Express API, you make standard HTTP requests using fetch (built into the browser) or a library like axios. The most common pattern is to fetch data inside a useEffect when a component first mounts. TypeScript lets you type the response data so your editor knows exactly what fields are available.

Fetching Typed Data from an Express API

import { useState, useEffect } from 'react';

interface Concert {
  _id: string;
  artist: string;
  venue: string;
  date: string;
}

function ConcertList() {
  const [concerts, setConcerts] = useState<Concert[]>([]);
  const [error, setError] = useState<string | null>(null);

  useEffect(() => {
    fetch("http://localhost:3000/api/concerts")
      .then((res: Response) => {
        if (!res.ok) throw new Error("Network response was not ok");
        return res.json();
      })
      .then((data: Concert[]) => setConcerts(data))
      .catch((err: Error) => setError(err.message));
  }, []);  // Empty array = fetch once on mount

  if (error) return <p>Error: {error}</p>;
  if (concerts.length === 0) return <p>Loading...</p>;

  return (
    <ul>
      {concerts.map((c: Concert) => (
        <li key={c._id}>{c.artist} — {c.venue}</li>
      ))}
    </ul>
  );
}

Managing Loading States with <Suspense>

React's built-in <Suspense> component lets you declaratively show a fallback UI (like a spinner or "Loading..." message) while a child component is waiting for asynchronous data. This keeps your loading logic clean and out of your component code.

import { Suspense, lazy } from 'react';

// Lazy-load a component (its code is only fetched when needed)
const ConcertList = lazy(() => import('./ConcertList.tsx'));

function App() {
  return (
    <Suspense fallback={<p>Loading concerts...</p>}>
      <ConcertList />
    </Suspense>
  );
}
Suspense works great with lazy loading. Use lazy() to split your app into smaller chunks. The code for a component is only downloaded when it's actually needed, making your initial page load faster.

📏 The Rules of React (How to Avoid Bugs)

Component Purity

A React component should be a pure function: given the same props and state, it should always return the same TSX. Components should not change any variables that existed before they were called (no mutating external objects), and they should not have side effects during rendering.

❌ Impure (Bad)

let count: number = 0;

function Counter() {
  // Mutating an external
  // variable during render!
  count++;
  return <p>{count}</p>;
}

✅ Pure (Good)

function Counter() {
  const [count, setCount] =
    useState<number>(0);
  // State is managed by React
  return <p>{count}</p>;
}

The Rules of Hooks

Hooks (useState, useEffect, etc.) have two strict rules. Breaking them will cause confusing, hard-to-debug errors.

Rule What It Means
Only call Hooks at the top level Never call a Hook inside a loop, an if statement, or a nested function. Hooks must always run in the same order on every render.
Only call Hooks from React functions Call Hooks from React function components or from custom Hooks — not from regular TypeScript functions.

❌ Wrong — Hook inside an if

function Bad() {
  if (loggedIn) {
    // ❌ Conditional Hook!
    const [name, setName] =
      useState<string>("");
  }
}

✅ Right — Hook at the top level

function Good() {
  const [name, setName] =
    useState<string>("");

  if (!loggedIn) return null;
  return <p>{name}</p>;
}

✨ React 19 Magic (The "Good to Know" Features)

The React Compiler

In older versions of React, developers had to manually optimize performance using techniques like useMemo, useCallback, and React.memo. These were confusing, easy to misuse, and added a lot of boilerplate. React 19 ships with a compiler that automatically analyzes your components and inserts these optimizations for you behind the scenes.

What this means for you: You don't have to learn advanced memoization techniques right away. Just write clean, standard React + TypeScript code, and the compiler will figure out the optimal way to re-render your components. Early adopters report 25–40% fewer unnecessary re-renders without changing a single line of code.

Easy Document Heads

Previously, if you wanted to change the page's <title> or add <meta> tags from inside a component, you needed a third-party library like React Helmet. In React 19, you can just drop these tags directly into your component's TSX, and React will automatically hoist them into the document's <head>.

Document Metadata in Components

function AboutPage() {
  return (
    <>
      {/* These are hoisted to the document <head> automatically */}
      <title>About Us | My App</title>
      <meta name="description" content="Learn about our team." />

      {/* Normal page content */}
      <h1>About Us</h1>
      <p>We build great things.</p>
    </>
  );
}
This is especially useful for SEO and multi-page apps. Each page component can declare its own title and meta tags, and React takes care of updating the document head whenever the user navigates. No extra libraries needed.

❌ Common Mistakes to Avoid

Wrong ❌ Right ✓
onClick={handleClick()} onClick={handleClick} — pass the function, don't call it
items.push(newItem) setItems([...items, newItem]) — never mutate state directly
Using class in TSX Use className — class is a reserved word in TS/JS
Calling a Hook inside an if Hooks must be at the top level of your component — every render, same order
Forgetting key on list items Always pass a unique key prop when using .map()
Using the array index as key Use a stable, unique ID from your data (e.g., _id from MongoDB)
Missing dependency in useEffect Include all referenced variables in the dependency array — your linter will warn you
Forgetting to check res.ok with fetch fetch doesn't throw on HTTP errors — always check the response status
Saving a component as .ts instead of .tsx Files that contain TSX markup must use the .tsx extension
Skipping the generic on useState(null) Use useState<User | null>(null) — otherwise TS thinks it's always null
