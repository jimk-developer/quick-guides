Tailwind CSS Core Concepts

Utility-First Styling, Responsive Design, Dark Mode, Theming, and Configuration — Tailwind v4.1+

🌊 Introduction

Tailwind CSS is a utility-first CSS framework that gives you low-level utility classes to build designs directly in your markup. Instead of writing custom CSS class names like .card-header or .sidebar-nav, you compose designs using small, single-purpose classes like p-4, bg-white, and text-lg.

The Utility-First Mindset

Traditional CSS encourages you to invent semantic class names and write separate stylesheets. Over time, these stylesheets grow, overlap, and become difficult to maintain. Tailwind flips this approach: styles live where your markup lives. Every element's appearance is clear at a glance — no hunting through external .css files to understand what a class does.

🎯 Utility-First

Small, composable classes that each do one thing — like mt-4 for margin-top or rounded-lg for border-radius.

📱 Mobile-First

Responsive prefixes (sm:, md:, lg:) apply styles at a given breakpoint and above.

🌙 Dark Mode Built-In

The dark: modifier makes theme adaptation effortless — no separate stylesheet required.

⚡ CSS-First Config (v4+)

Starting with v4, Tailwind is configured directly in CSS using @theme — no more JavaScript config file required.

This guide covers the essential concepts you need to be productive with Tailwind CSS v4.1 and later. All examples use React with TypeScript (TSX) unless noted otherwise, and use generic placeholder data throughout.

🎨 Styling with Utility Classes

The core idea behind Tailwind is that you build your UI by applying pre-existing utility classes directly to HTML or JSX elements. Each class maps to a single CSS property — p-4 sets padding: 1rem, bg-white sets background-color: white, rounded-xl sets a large border-radius, and so on.

This approach improves maintainability because you never need to invent class names, and you never end up with bloated stylesheets full of unused rules. Every class you see in your markup is doing exactly one thing, and removing an element removes its styles too.

Example: A Styled Card Component (TSX)

// ProductCard.tsx
interface ProductCardProps {
  title: string;
  description: string;
  price: number;
}

export default function ProductCard({ title, description, price }: ProductCardProps) {
  return (
    <div className="max-w-sm rounded-xl bg-white p-6 shadow-lg">
      <h2 className="mb-2 text-xl font-bold text-gray-900">
        {title}
      </h2>
      <p className="mb-4 text-sm text-gray-600">
        {description}
      </p>
      <span className="inline-block rounded-full bg-cyan-100 px-3 py-1 text-sm font-semibold text-cyan-800">
        ${price.toFixed(2)}
      </span>
    </div>
  );
}
No custom CSS was written. The card's padding, border-radius, shadow, typography, and colors are all handled by utility classes. If you later remove this component, zero leftover CSS remains in your project.

🖱️ Hover, Focus, and Other States

Tailwind provides state modifiers that let you conditionally apply styles based on user interaction — without writing any custom CSS pseudo-class rules. You simply prefix a utility class with the state name and a colon.

Modifier CSS Equivalent Example
hover: :hover hover:bg-blue-700
focus: :focus focus:outline-none focus:ring-2
active: :active active:scale-95
disabled: :disabled disabled:opacity-50
first: :first-child first:rounded-t-lg

Example: Interactive Submit Button (TSX)

// SubmitButton.tsx
export default function SubmitButton() {
  return (
    <button
      type="submit"
      className="rounded-lg bg-blue-600 px-6 py-3 font-semibold text-white
                 transition-all duration-200
                 hover:bg-blue-700 hover:shadow-lg
                 focus:outline-none focus:ring-4 focus:ring-blue-300
                 active:scale-95 active:bg-blue-800
                 disabled:cursor-not-allowed disabled:opacity-50"
    >
      Submit Form
    </button>
  );
}
Modifiers are stackable. You can combine them like sm:hover:bg-blue-700 to apply a hover effect only on small screens and above, or dark:hover:bg-blue-400 for dark-mode-specific hover colors.

📱 Responsive Design

Tailwind uses a mobile-first breakpoint system. Un-prefixed utilities apply to all screen sizes, and breakpoint prefixes apply at that breakpoint and above. This means you design for small screens first, then layer on styles for larger screens.

Prefix Minimum Width CSS Media Query
sm: 640px @media (min-width: 640px)
md: 768px @media (min-width: 768px)
lg: 1024px @media (min-width: 1024px)
xl: 1280px @media (min-width: 1280px)
2xl: 1536px @media (min-width: 1536px)

Example: Responsive Grid Layout (TSX)

// FeatureGrid.tsx
const features = [
  { id: 1, label: "Fast Builds", detail: "Incremental compilation in microseconds." },
  { id: 2, label: "Responsive", detail: "Mobile-first breakpoints out of the box." },
  { id: 3, label: "Customizable", detail: "CSS-first config with @theme." },
];

export default function FeatureGrid() {
  return (
    {/* Single column on mobile → 2 cols on md → 3 cols on lg */}
    <div className="grid grid-cols-1 gap-6 md:grid-cols-2 lg:grid-cols-3">
      {features.map((feat) => (
        <div
          key={feat.id}
          className="rounded-lg border border-gray-200 bg-white p-6 shadow-sm"
        >
          <h3 className="mb-2 text-lg font-bold">{feat.label}</h3>
          <p className="text-sm text-gray-600">{feat.detail}</p>
        </div>
      ))}
    </div>
  );
}

Key Insight: Mobile-First Means "Work Upward"

The un-prefixed grid-cols-1 establishes a single column layout for all screens. Then md:grid-cols-2 kicks in at 768px, and lg:grid-cols-3 at 1024px. You never need to write a max-width media query — if you don't override a breakpoint, the previous style simply carries through.

🌙 Dark Mode

Tailwind includes a built-in dark: modifier that lets you style elements differently when dark mode is active. By default in v4, the dark: variant is driven by the user's operating system preference via prefers-color-scheme — no configuration needed.

Strategy Options

Strategy How It Works CSS Setup
Media (default) Follows the OS preference automatically No setup required — works out of the box
Selector / Class Toggle via a .dark class on <html> @custom-variant dark (&:is(.dark *));
Data Attribute Toggle via a data-theme="dark" attribute @custom-variant dark (&:where([data-theme=dark], [data-theme=dark] *));

Example: Dashboard Widget with Dark Mode (TSX)

// StatsWidget.tsx
interface StatsWidgetProps {
  label: string;
  value: string;
  change: string;
}

export default function StatsWidget({ label, value, change }: StatsWidgetProps) {
  return (
    <div className="rounded-xl bg-white p-6 shadow-md
                 dark:bg-gray-800 dark:shadow-gray-900/30">
      <p className="text-sm font-medium text-gray-500
                 dark:text-gray-400">
        {label}
      </p>
      <p className="mt-1 text-3xl font-bold text-gray-900
                 dark:text-white">
        {value}
      </p>
      <p className="mt-2 text-sm font-semibold text-green-600
                 dark:text-green-400">
        {change}
      </p>
    </div>
  );
}
Setting up class-based toggling in v4. If you want manual control, add @custom-variant dark (&:is(.dark *)); to your main CSS file alongside the Tailwind import. Then toggle the .dark class on <html> with JavaScript, saving the preference to localStorage.

🎛️ Theme Variables

In Tailwind CSS v4 and later, customization is done directly in CSS using the @theme directive — no tailwind.config.js file needed. Values you define inside @theme become CSS custom properties and are automatically available as Tailwind utility classes.

Defining Theme Variables (CSS)

/* app.css */
@import "tailwindcss";

@theme {
  /* Custom spacing */
  --spacing-nav: 4.5rem;
  --spacing-header: 3.5rem;

  /* Custom fonts */
  --font-display: "Inter", "Helvetica Neue", sans-serif;
  --font-mono: "Fira Code", "Consolas", monospace;

  /* Custom breakpoint */
  --breakpoint-3xl: 1920px;

  /* Brand colors */
  --color-brand-500: #0891B2;
  --color-brand-600: #0E7490;
}

Example: Using Theme Variables in a Nav Bar (TSX)

// NavBar.tsx — uses the custom theme values defined above
export default function NavBar() {
  return (
    <nav className="flex h-header items-center bg-brand-600 px-nav font-display">
      <span className="text-lg font-bold text-white">
        My App
      </span>
      <div className="ml-auto flex gap-6">
        <a href="/dashboard" className="text-white/80 hover:text-white">Dashboard</a>
        <a href="/settings" className="text-white/80 hover:text-white">Settings</a>
      </div>
    </nav>
  );
}

How It Connects

The variable --spacing-nav: 4.5rem defined in @theme creates the utility px-nav. The variable --spacing-header: 3.5rem creates h-header. The variable --font-display creates font-display. And --color-brand-600 creates bg-brand-600, text-brand-600, etc. — all automatically.

🎨 Colors

Tailwind ships with an extensive default color palette organized into shades from 50 (lightest) through 950 (darkest). You apply colors using the pattern {property}-{color}-{shade}, for example bg-blue-500 or text-gray-700.

Opacity Modifiers

Tailwind v4 uses color-mix() under the hood. You can apply opacity directly with a slash modifier: bg-blue-500/75 renders blue-500 at 75% opacity. This works on any color utility — backgrounds, text, borders, shadows, and more.

Custom Brand Colors

You can define custom brand colors in your @theme block and then use them just like built-in colors:

Example: Task List with Color Highlights (TSX)

// TaskList.tsx
interface Task {
  id: number;
  name: string;
  isActive: boolean;
}

const tasks: Task[] = [
  { id: 1, name: "Review pull request", isActive: true },
  { id: 2, name: "Update documentation", isActive: false },
  { id: 3, name: "Deploy staging build", isActive: false },
];

export default function TaskList() {
  return (
    <ul className="space-y-2">
      {tasks.map((task) => (
        <li
          key={task.id}
          className={`rounded-lg px-4 py-3 text-sm font-medium ${
            task.isActive
              ? "border-l-4 border-cyan-500 bg-cyan-50 text-cyan-900"
              : "bg-gray-50 text-gray-700"
          }`}
        >
          {task.name}
        </li>
      ))}
    </ul>
  );
}

Plain HTML Version

The same pattern works identically outside React — in a standard HTML file:

Alternative: Task List in Plain HTML

<ul class="space-y-2">
  <li class="rounded-lg border-l-4 border-cyan-500 bg-cyan-50 px-4 py-3 text-sm font-medium text-cyan-900">
    Review pull request
  </li>
  <li class="rounded-lg bg-gray-50 px-4 py-3 text-sm font-medium text-gray-700">
    Update documentation
  </li>
  <li class="rounded-lg bg-gray-50 px-4 py-3 text-sm font-medium text-gray-700">
    Deploy staging build
  </li>
</ul>
Opacity on any color. Try bg-blue-500/50 for a 50% transparent blue, or text-gray-900/75 for slightly faded text. In v4, this compiles to color-mix(in oklab, ...) for accurate color blending.

🔧 Adding Custom Styles

Sometimes utility classes alone aren't enough. Tailwind provides escape hatches to handle one-off values or integrate standard CSS alongside the utility system.

Arbitrary Values

If Tailwind doesn't have a utility for a specific value, you can use bracket notation to supply any CSS value directly:

Using Arbitrary Values in Markup

<!-- Position an element exactly 117px from the top -->
<div className="absolute top-[117px] left-[2.5rem] w-[calc(100%-3rem)]">
  Precisely positioned content
</div>

<!-- Use an arbitrary color -->
<span className="text-[#1DA1F2]">Custom brand blue</span>

<!-- Arbitrary font size -->
<h1 className="text-[2.75rem] leading-[1.2]">Big headline</h1>

Custom CSS with @theme and @layer

For styles that go beyond one-off values, you can extend Tailwind in your CSS file using @theme for new design tokens or @layer to inject styles into Tailwind's cascade layers:

Custom CSS in Your Stylesheet

@import "tailwindcss";

@theme {
  /* Add a custom color to the theme */
  --color-corporate: #1B3A5C;

  /* Add a custom spacing value */
  --spacing-18: 4.5rem;
}

/* Add base styles to Tailwind's base layer */
@layer base {
  h1 {
    font-size: 2.25rem;
    font-weight: 700;
    line-height: 1.2;
  }
}

/* Add a custom component style */
@layer components {
  .card {
    background: white;
    border-radius: var(--radius-lg);
    padding: var(--spacing-6);
    box-shadow: var(--shadow-md);
  }
}
Use escape hatches sparingly. The power of Tailwind comes from staying within the utility system. Arbitrary values and custom CSS are perfectly fine for one-off situations, but if you find yourself repeating the same arbitrary value, consider adding it to @theme instead.

🔍 Detecting Classes in Source Files

Tailwind's compiler scans your source files (.tsx, .html, .jsx, .vue, etc.) to find which utility classes you've used, and generates a CSS file containing only those styles. Classes that never appear in your code are never included in the output — keeping your CSS bundle small.

In v4, Tailwind automatically detects your source files using smart heuristics. It ignores files in .gitignore and skips binary files. If you need to include an additional source, use the @source directive:

/* Tell Tailwind to also scan files in a specific directory */
@source "../shared-components";

The Class Detection Gotcha

The compiler works by scanning for complete class strings. If you dynamically build class names using string concatenation, the compiler won't find them and they'll be purged from the output.

❌ Dynamically Constructed (Broken)

// The compiler cannot detect these!
const color = "red";
const shade = "500";

<div className={`bg-${color}-${shade}`}>
  Oops — this will be unstyled!
</div>

✅ Complete Strings (Works)

// Full class names ARE detected
const colorClass =
  isError ? "bg-red-500" : "bg-green-500";

<div className={colorClass}>
  This works perfectly!
</div>
Rule of thumb: Always use complete, unbroken class name strings. Never build Tailwind class names by concatenating partial strings — map conditions to full class names instead.

📐 Functions and Directives

Tailwind provides several CSS-level directives and functions that let you work with the framework at a deeper level. These are used inside your CSS files, not in markup.

Directive / Function Purpose
@import "tailwindcss" The single entry point for Tailwind in v4 — imports base, components, and utilities layers.
@theme { ... } Defines design tokens (colors, spacing, fonts, breakpoints) as CSS custom properties.
@apply Inlines Tailwind utility classes into a custom CSS rule. Useful for styling elements you don't control.
@layer base / components / utilities Injects custom CSS into one of Tailwind's cascade layers for correct specificity ordering.
@source Tells the compiler to scan an additional directory or glob pattern for class names.
@custom-variant Defines a custom variant (e.g., overriding the dark: variant for class-based toggling).
theme() References a theme value inside CSS. In v4 you can often use var() instead, but theme() is kept for compatibility.

Using @apply for Reusable Styles

The @apply directive lets you extract a set of utility classes into a single CSS rule. This is especially helpful for elements you don't control directly — such as third-party components or elements generated by a CMS.

Example: Extracting a Reusable Button Style (CSS)

@import "tailwindcss";

/* Extract a reusable .btn class from utility classes */
@layer components {
  .btn {
    @apply inline-flex items-center justify-center rounded-lg px-5 py-2.5
           text-sm font-semibold transition-colors duration-200;
  }

  .btn-primary {
    @apply btn bg-blue-600 text-white hover:bg-blue-700
           focus:outline-none focus:ring-4 focus:ring-blue-300;
  }

  .btn-outline {
    @apply btn border border-gray-300 bg-transparent text-gray-700
           hover:bg-gray-50 focus:outline-none focus:ring-4 focus:ring-gray-200;
  }
}

Now you can use <button class="btn-primary">Save</button> anywhere in your project, and the styling is pulled from a centralized definition that's still powered by Tailwind utilities.

Don't overuse @apply. The Tailwind team recommends using it only when you cannot apply utility classes directly — such as third-party elements or CMS-generated content. If you can use utilities in your markup, that's almost always the better choice.

❌ Common Mistakes to Avoid

Wrong ❌ Right ✓
bg-${color}-500 Use complete class strings: "bg-red-500" or "bg-green-500"
Writing a separate .card { ... } stylesheet Apply utilities directly in markup: className="rounded-lg p-6 shadow"
sm:grid-cols-1 lg:grid-cols-3 without the un-prefixed base Start with grid-cols-1 then add lg:grid-cols-3 — mobile-first!
Using tailwind.config.js in a v4 project Use the @theme directive in your CSS — v4 is CSS-first by default
@tailwind base; @tailwind components; @tailwind utilities; In v4, just @import "tailwindcss"; — one line replaces the three directives
Applying dark: classes without testing both themes Always verify components in both light and dark mode during development
Creating deeply nested @apply chains Use @apply sparingly — prefer utilities in markup wherever possible
Forgetting transition with hover/focus effects Add transition-colors or transition-all for smooth state changes

🚀 Conclusion & Next Steps

Tailwind CSS transforms the way you style web applications. Instead of maintaining separate stylesheets that grow out of control, you compose designs right in your markup using a powerful, consistent set of utility classes. Here's what we covered:

🎯 Utility-First

Build layouts using small, composable classes applied directly in your HTML or JSX.

🖱️ State Modifiers

Use hover:, focus:, active:, and more to handle interactivity without custom CSS.

📱 Responsive Design

Mobile-first breakpoint prefixes let you progressively enhance layouts for larger screens.

🌙 Dark Mode

The dark: modifier adapts your UI to dark themes via OS preferences or class-based toggling.

🎛️ Theme Variables

The @theme directive lets you define a custom design system right in CSS.

📐 Directives & Functions

@apply, @layer, and @source give you fine-grained control when you need it.

Best Practices for Scaling Tailwind

Tips for Larger Applications

Extract repeated patterns into components, not CSS classes. In frameworks like React, create reusable components (e.g., <Button>, <Card>) that encapsulate their Tailwind classes. This is far more maintainable than creating dozens of @apply-based CSS classes.

Keep your @theme block organized. Group tokens by category — spacing, colors, typography, breakpoints — and add comments. This serves as your project's design system documentation.

Use consistent naming conventions. If you add custom colors, follow Tailwind's shade pattern (brand-50 through brand-950) so the API feels familiar to your team.

Leverage a class-sorting tool. The official Prettier plugin for Tailwind CSS automatically sorts utility classes in a consistent, predictable order — reducing merge conflicts and making code review easier.
